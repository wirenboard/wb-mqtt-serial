# СТП 001-2025 «Шаблоны устройств для драйвера wb-mqtt-serial»

**Редакция 03. Черновик. Дата: 2026-05-25.**

> Настоящая редакция замещает изд.01 и изд.02 (внутренние черновики).
>
> Документ содержит только правила, специфичные для шаблонов wb-mqtt-serial:
>
> - правила формирования идентификаторов MQTT-топиков — в [СТП 002-2026](СТП%20002-2026%20\(MQTT-топики,%20черновик\).md);
> - правила оформления UI-текстов и переводов — в [СТП 003-2026](СТП%20003-2026%20\(UI-тексты,%20черновик\).md);
> - правила публичных MQTT-соглашений — в [Wiren Board MQTT Conventions](https://github.com/wirenboard/conventions).
>
> Документ согласован с указанными источниками и не противоречит им.

---

## 1. Область применения

Стандарт устанавливает правила формирования шаблонов устройств для драйвера `wb-mqtt-serial` Wiren Board. Шаблон — JSON-файл, описывающий устройство, его параметры настройки и контролы, публикуемые драйвером в MQTT.

Стандарт применяется ко всем новым шаблонам устройств Wiren Board. К партнёрским и пользовательским шаблонам — рекомендательно.

Существующие шаблоны мигрируются постепенно по отдельному плану миграции.

---

## 2. Принципы

Шаблон должен быть:

- **однозначно интерпретируемым** драйвером, валидатором и линтером;
- **понятным конечному пользователю** в веб-конфигураторе (вместе со стилем по СТП 003-2026);
- **миграционно совместимым** — при выпуске новой версии устройства имена сохраняемых сущностей не меняются (см. §10).

---

## 3. Структура шаблона

Шаблон — JSON-документ, соответствующий JSON-схеме `wb-mqtt-serial-device-template.schema.json`. Схема определяет обязательные и допустимые поля, типы значений и базовую структуру.

Настоящий стандарт устанавливает требования сверх схемы — содержательные правила формирования значений, канонические имена и стиль.

**Верхний уровень шаблона:**

```jsonc
{
  "device_type": "...",         // обязательно, см. §4
  "title": "...",               // отображаемое имя модели в каталоге устройств
  "group": "...",               // идентификатор группы устройств в каталоге (см. groups.json)
  "hw": [ ... ],                // соответствие физическому устройству (Modbus signature)
  "device": {
    "name": "...",              // отображаемое имя в UI (см. §4.3)
    "id": "...",                // идентификатор устройства, см. СТП 002-2026
    "translations": { ... },    // переводы (см. СТП 003-2026)
    "groups":     [ ... ],      // см. §5
    "parameters": [ ... ],      // см. §6
    "channels":   [ ... ],      // см. §7
    "setup":      [ ... ],      // опционально, см. §9.6
    "subdevices": [ ... ]       // опционально, см. §9.7
  }
}
```

Поля `device.setup`, `device.subdevices`, `device.translations`, `hw` — опциональны.

## 4. Идентификаторы, стиль текстов и канонические имена

### 4.1 Идентификаторы

Правила формирования идентификаторов (имён контролов, параметров, групп, `device.id`) определяются СТП 002-2026 «Идентификаторы MQTT-топиков в экосистеме Wiren Board».

Кратко: алфавит `[a-z0-9_]`, snake_case, не более четырёх слов и цифр, не более 64 символов, регулярное выражение `^[a-z][a-z0-9]*(_[a-z0-9]+)*$`.

### 4.2 Стиль текстов

Правила оформления текстов, видимых пользователю (`title`, `description`, значения `enum_titles`), определяются СТП 003-2026 «Стиль текстов в пользовательском интерфейсе Wiren Board».

Кратко: ASCII-кавычки и дефис; «е» вместо «ё»; точка как десятичный разделитель; EN — Title Case, RU — Sentence case; аббревиатуры заглавными; единицы в круглых скобках после названия и не в id; описания — короткие, без точки в конце.

### 4.3 `device.name`

Поле `device.name` — литеральное отображаемое имя устройства в UI и в MQTT-метаключе `name` устройства. Не является ключом перевода. Если требуется многоязычность — используется поле `device.title` (объект `{en, ru, ...}`) и/или `device.translations`.

Имя должно содержать обозначение модели (например, `WB-MR6C v.3`, `WB-MAP12H v.3`); регистр и пунктуация — как на маркировке устройства.

### 4.4 `device_type`

`device_type` — машинный идентификатор семейства шаблона, используется драйвером и веб-конфигуратором для привязки к подсистеме (`device_type` указывается в конфигурации `wb-mqtt-serial.conf`).

Формирование `device_type` — по правилам идентификаторов СТП 002-2026 + допускается заглавная и дефис в составе бренда (например, `WB-MR6CV3`, `EKF-DDS353B`), поскольку это идентификатор семейства, а не топик.

### 4.5 Канонические имена сущностей

Идентификаторы и переводы для сущностей, многократно встречающихся в WB-шаблонах (служебные каналы, стандартные группы, входы/выходы, счётчики, трёхфазные величины), определяются Приложениями А–Г настоящего стандарта.

Если сущность присутствует в приложениях — следует использовать каноническую форму, а не свою.

## 5. Группы (`groups[]`)

Группы определяют иерархию отображения параметров и каналов в UI веб-конфигуратора.

### 5.1 Иерархия и префиксы

Допускается до трёх уровней вложенности. Префикс идентификатора группы маркирует уровень:

| Уровень | Префикс | Пример |
| --- | --- | --- |
| Корневой | `g_` | `g_general`, `g_hw_info`, `g_in1` |
| Второй | `gg_` | `gg_in1_channels`, `gg_in1_press_params` |
| Третий | `ggg_` | `ggg_in1_out1_button_actions` |

### 5.2 Поля группы

```jsonc
{
  "id":       "g_general",
  "title":    "General",                            // или объект {en, ru}
  "group":    null | "<parent_group_id>",           // null для корневой
  "ui_options": { "wb": { "disable_title": false } }
}
```

Поле `title` — отображаемое имя группы в UI (по правилам СТП 003-2026).

### 5.3 Канонические корневые группы

См. Приложение Б. При наличии соответствующей сущности её следует помещать в каноническую группу.

## 6. Параметры (`parameters[]`)

Параметры — поля настройки устройства, выставляемые пользователем в веб-конфигураторе при создании конфигурации (записываются в устройство через `setup` или при назначении).

### 6.1 Основные поля

```jsonc
{
  "id":          "baud_rate",
  "title":       "Baud Rate",                       // или объект {en, ru}
  "description": "...",                             // опционально, по правилам СТП 003-2026
  "address":     "...",                             // см. §8
  "reg_type":    "holding",                         // см. §8
  "format":      "u16",                             // см. §8
  "default":     9600,
  "enum":        [9600, 19200, 38400, 57600, 115200],
  "enum_titles": ["9600", "19200", ...],            // или объект {en, ru}
  "group":       "g_general",
  "required":    true | false,
  "condition":   "...",                             // см. §9.3
  "fw":          ">=1.23.0"                         // см. §9.4
}
```

### 6.2 Идентификаторы параметров

Для `parameters[].id` допустимы три формы:

| Форма | Когда применять | Пример |
| --- | --- | --- |
| **Описательный snake_case** | Для устройств с малым числом параметров (≤ ~20), где смысл выражается словом. Преобладающая форма в шаблонах WB | `baud_rate`, `safety_timer_s`, `disable_indication` |
| **`p<N>` — позиционный** | Для устройств с десятками параметров без естественных имён (контроллеры, инверторы, реле в общем стиле производителя) | `p1`, `p2`, …, `p47` |
| **Panel-код** | Для устройств с этикеточными кодами параметров (частотные преобразователи, ПЛК), где код — общеизвестная сущность | `f0_04`, `pn_01_03`, `a01` |

В пределах одного шаблона допускается ровно одна форма.

### 6.3 Нормализация panel-кодов

Panel-код устройства часто содержит точку или дефис (например, `F0.04`, `Pn-01.03`). В id параметра он приводится к нормализованной форме:

- символы `.`, `-`, `:` заменяются на `_`;
- буквы приводятся к нижнему регистру;
- ведущие нули не убираются.

Примеры:

| Panel-код на устройстве | id в шаблоне |
| --- | --- |
| `F0.04` | `f0_04` |
| `Pn-01.03` | `pn_01_03` |
| `A01` | `a01` |

В `title`/`description` panel-код сохраняется в исходной маркировке: `title: "F0.04 — Frequency setting"`.

### 6.4 Типы значений

| Тип параметра | Признак |
| --- | --- |
| Целое число | `format: u8 | s8 | u16 | s16 | u32 | s32 | u64 | s64` |
| Число с плавающей точкой | `format: float | double` |
| Перечислимое | поле `enum`, опционально `enum_titles` |
| Логическое (двоичное) | `format: u8` + `enum: [0, 1]` + `enum_titles` |
| BCD | `format: bcd16 | bcd32` |
| Строка | `format: string` + `string_data_size` |

### 6.5 Группа параметра

`parameters[].group` — id группы (см. §5), в которой параметр отображается. Если не указано — корень.

## 7. Каналы (`channels[]`)

Каналы — контролы устройства, публикуемые драйвером в MQTT и отображаемые пользователю в реальном времени.

### 7.1 Основные поля

```jsonc
{
  "name":      "Input 1",                       // имя MQTT-топика, см. СТП 002-2026
  "type":      "switch",                        // тип контрола, см. §7.2
  "reg_type":  "discrete",                      // см. §8
  "address":   12,                              // см. §8
  "format":    "u16",                           // см. §8
  "scale":     0.01,                            // опционально
  "offset":    0,                               // опционально
  "units":     "V",                             // отображаемые единицы (см. СТП 003-2026)
  "min":       0,
  "max":       250,
  "precision": 0.01,
  "readonly":  true | false,
  "sporadic":  true,                            // см. §7.3
  "enabled":   false,                           // канал по умолчанию выключен
  "error_value": 65535,
  "group":     "gg_in1_channels",
  "condition": "...",                           // см. §9.3
  "fw":        ">=1.20.0",                      // см. §9.4
  "write_address": ...                          // см. §9.5
}
```

### 7.2 Типы контролов

Типы — из конвенции Wiren Board MQTT Conventions §3 (значения метаключа `meta/type`):

`switch`, `pushbutton`, `range`, `value`, `text`, `rgb`, `wo-switch`, `temperature`, `voltage`, `current`, `power`, `pressure`, `concentration`, `lux`, `sound_level`, `alarm`, `unixtime`, `w1-id` и иные, определяемые конвенцией.

Тип следует выбирать по смыслу величины. Для физических величин предпочтительно использовать тип-синоним (`temperature`, `voltage`, …), а не общий `value` — это упрощает рендеринг в UI и интеграцию с Home Assistant.

### 7.3 Спорадические каналы

Поле `sporadic: true` указывает драйверу не опрашивать канал периодически; значение публикуется только при изменении. Применяется к редко изменяющимся сущностям: счётчикам нажатий, итоговым счётчикам потребления, событиям.

### 7.4 Группа канала

`channels[].group` — id группы (см. §5), в которой канал отображается.

## 8. Адресация и форматы значений

### 8.1 `reg_type`

Допустимые значения определяются JSON-схемой. Типичные:

| Значение | Назначение |
| --- | --- |
| `holding` | Modbus holding registers (читать/писать) |
| `input` | Modbus input registers (только чтение) |
| `coil` | Modbus coils (битовые, читать/писать) |
| `discrete` | Modbus discrete inputs (битовые, только чтение) |
| `press_counter` | Регистр счётчика нажатий (WB-специфика) |

### 8.2 `address`

Адрес регистра, число или строка. Поддерживаемые формы:

- целое число: `123`;
- шестнадцатеричное в строке: `"0x7B"`;
- битовое: `"<reg>:<bit_offset>"` или `"<reg>:<bit_offset>:<bit_width>"` (см. §9.1).

### 8.3 `format`

| Значение | Размер регистров |
| --- | --- |
| `u8`, `s8` | 1 |
| `u16`, `s16` | 1 |
| `u32`, `s32`, `float`, `bcd32` | 2 |
| `u64`, `s64`, `double` | 4 |
| `string` | определяется `string_data_size` |

### 8.4 `word_order`

Для многорегистровых форматов: `big_endian` (по умолчанию, MSB first) или `little_endian` (LSB first). Уточняется по даташиту устройства.

### 8.5 `scale` и `offset`

Преобразование сырого значения регистра в физическую величину:

```
displayed = raw * scale + offset
```

`scale` — обычно степень десяти (`0.1`, `0.01`, `0.001`), реже произвольное.

## 9. Специальные случаи

### 9.1 Битовые поля

Битовый канал/параметр имеет адрес вида `"<reg>:<bit_offset>"` (один бит) или `"<reg>:<bit_offset>:<bit_width>"` (поле из нескольких бит):

```jsonc
{ "name": "Alarm 1", "reg_type": "input", "address": "0x100:5",   "type": "alarm" }
{ "name": "Error",   "reg_type": "input", "address": "0x100:0:4", "type": "value" }
```

Bit offset считается от младшего бита. Поле не должно пересекать границы регистра.

### 9.2 Многорегистровые параметры

Для `format` размером > 1 регистра поле `address` указывает первый регистр. Порядок слов определяется `word_order`.

```jsonc
{
  "name":       "Energy Total",
  "reg_type":   "input",
  "address":    1024,
  "format":     "u32",
  "word_order": "big_endian",
  "type":       "energy_power"
}
```

### 9.3 Условия (`condition`)

Поле `condition` — JavaScript-выражение, ссылающееся на id параметров. Если выражение возвращает `false`, элемент не активен (скрывается, не опрашивается, не записывается).

```jsonc
{
  "name":      "Voltage L2",
  "address":   200,
  "type":      "voltage",
  "condition": "phases == 3"
}
```

Идентификаторы в `condition` — id параметров (не каналов и не топиков); должны существовать в `parameters[]` шаблона. Сравнения, логические операторы, числовые литералы — стандартный JS.

### 9.4 Зависимость от прошивки (`fw`)

Поле `fw` — семантическая версия (semver) или диапазон, требуемая для активности элемента. Если прошивка устройства ниже — элемент скрывается.

```jsonc
{ "name": "MCU Voltage", "fw": ">=1.20.0", ... }
```

### 9.5 Разделённый адрес записи (`write_address`)

Применяется, когда регистр имеет разные адреса для чтения и записи. Также используется для нестандартных бинарных значений в паре с `on_value` / `off_value`:

```jsonc
{
  "name":          "Relay 1",
  "reg_type":      "holding",
  "address":       100,
  "write_address": 200,
  "type":          "switch"
}
```

```jsonc
{
  "name":      "Door Lock",
  "reg_type":  "holding",
  "address":   50,
  "on_value":  "0x00a5",
  "off_value": "0x005a",
  "type":      "switch"
}
```

### 9.6 Setup-операции (`setup[]`)

Массив операций, выполняемых драйвером при подключении устройства (запись начальных значений в регистры). Используется для устройств, требующих инициализации.

```jsonc
"setup": [
  { "title": "Set Modbus delay", "address": 200, "reg_type": "holding", "value": 5, "fw": ">=1.0.0" }
]
```

### 9.7 Subdevices (`subdevices[]`)

Описание под-устройств (логических компонентов) внутри одного физического устройства. Каждое subdevice имеет собственные `parameters[]`, `channels[]` и адресное смещение.

Применяется для модульных устройств — например, `WB-MWAC v.2` (модуль учёта воды и др.) и интеграций многоканальных счётчиков.

Подробности структуры — в JSON-схеме.

## 10. Имя файла и версионирование шаблонов

### 10.1 Имя файла

Имя файла шаблона: `config-<device_type>.json.jinja` для Jinja-шаблонов или `config-<device_type>.json` для статических JSON.

`<device_type>` — точное значение `device_type` шаблона в нижнем регистре с заменой запрещённых символов (см. §4.4) на `-`.

Примеры:

- `device_type: "WB-MR6CV3"` → `config-wb-mr6cv3.json.jinja`
- `device_type: "EKF-DDS353B"` → `config-ekf-dds353b.json.jinja`

### 10.2 Версионирование шаблона

При выпуске устройства новой ревизии (отличающейся регистровой картой, появлением/исчезновением сущностей) применяется один из подходов:

**1. Совместимая эволюция (предпочтительно).** Изменения в существующем шаблоне с использованием `condition`, `fw` для управления видимостью элементов. Имена сохраняемых сущностей не меняются (см. §11 СТП 002-2026 «Совместимость со старыми именами»).

**2. Несовместимая ревизия.** Создаётся новый шаблон с другим `device_type` (например, `WB-MR6CV3` → `WB-MR6CV4`); старый шаблон при этом помечается как deprecated.

### 10.3 Deprecated-шаблоны

Шаблон, заменённый новой версией, помечается как deprecated:

- в имя файла добавляется суффикс `-deprecated`: `config-<device_type>-deprecated.json.jinja`;
- в верхнем уровне шаблона устанавливается `"deprecated": true`;
- драйвер продолжает обрабатывать существующие конфигурации, но шаблон не предлагается для новых устройств в каталоге.

# Приложения (нормативные)

## Приложение А. Канонические служебные каналы

Служебные каналы — каналы устройств Wiren Board, общие для большинства семейств; размещаются в группе `g_hw_info` (см. Приложение Б).

| Идентификатор (`name`) | EN (`title`) | RU | Назначение |
| --- | --- | --- | --- |

## Приложение В. Канонические каналы цифровых входов и выходов

### В.1 Дискретные входы

| Идентификатор (`name`) | EN | RU | Назначение |
| --- | --- | --- | --- |
| `input_<N>` | Input N | Вход N | Текущее состояние дискретного входа N |

`<N>` — номер входа (`input_1`, `input_2`, …).

### В.2 Релейные выходы

| Идентификатор (`name`) | EN | RU | Назначение |
| --- | --- | --- | --- |
| `k<N>` | K N | K N | Релейный выход N (по этикетке на устройстве) |
| `relay_<N>` | Relay N | Реле N | Альтернатива для устройств без буквенной маркировки |

Для устройств Wiren Board с маркировкой `K1`, `K2`, … применяется `k<N>`.

### В.3 Счётчики нажатий

Для каждого входа `<N>`, поддерживающего распознавание нажатий, определяются следующие счётчики:

| Идентификатор (`name`) | EN | RU |
| --- | --- | --- |
| `input_<N>_counter` | Input N Counter | Вход N: Счётчик нажатий |
| `input_<N>_single_press_counter` | Input N Single Press Counter | Вход N: Счётчик одиночных нажатий |
| `input_<N>_long_press_counter` | Input N Long Press Counter | Вход N: Счётчик длинных нажатий |
| `input_<N>_double_press_counter` | Input N Double Press Counter | Вход N: Счётчик двойных нажатий |
| `input_<N>_shortlong_press_counter` | Input N Shortlong Press Counter | Вход N: Счётчик коротких + длинных нажатий |
| `reset_all_counters` | Reset All Counters | Сбросить все счётчики |

**Примечания:**

- Употребляется «Single Press», не «Short Press» (по противопоставлению с «Long» и «Double»).
- «Shortlong» пишется слитно (как в id, так и в `title`).
- Все счётчики имеют `reg_type: "press_counter"`, `sporadic: true`.

## Приложение Г. Канонические трёхфазные электрические величины

Применяются для счётчиков энергии и измерителей мощности.

### Г.1 Однофазные/трёхфазные счётчики

| Идентификатор (`name`) | EN | RU | Единицы |
| --- | --- | --- | --- |
| `frequency` | Frequency | Частота сети | Hz |
| `urms_l<N>` | Urms L*N* | Напряжение фазы *N* | V |
| `irms_l<N>` | Irms L*N* | Ток фазы *N* | A |
| `p_l<N>` | P L*N* | Активная мощность фазы *N* | W |
| `q_l<N>` | Q L*N* | Реактивная мощность фазы *N* | var |
| `s_l<N>` | S L*N* | Полная мощность фазы *N* | VA |
| `pf_l<N>` | PF L*N* | Коэффициент мощности фазы *N* | — |
| `voltage_angle_l<N>` | Voltage Angle L*N* | Угол напряжения фазы *N* | ° |
| `ap_energy_l<N>` | AP Energy L*N* | Активная энергия (+) фазы *N* | kWh |
| `an_energy_l<N>` | AN Energy L*N* | Активная энергия (−) фазы *N* | kWh |

| `thdn_i_l<N>` | THDN I L*N* | THDN тока фазы *N* | % |

`<N>` — номер фазы (`1`, `2`, `3`).

### Г.2 Многоканальные счётчики (MAP-серия)

Для счётчиков, измеряющих несколько физических вводов (12-канальные и т. п.), используется двухуровневая нотация: `channel<K>_<величина>_l<N>`.

Примеры: `channel1_urms_l1`, `channel1_p_l2`, `channel4_ap_energy_l3`.

Группировка — в `channel<K>_l<N>` (фазная подгруппа канала) и `channel<K>_total` (итог по каналу).

### Г.3 Соответствие OBIS

Канонические имена сопоставляются с OBIS-кодами по Приложению Д. OBIS-код приводится в поле `description` канала.

## Приложение Д. OBIS-коды для счётчиков энергии

OBIS (IEC 62056-6-1) — стандартизованная нотация кодов для счётчиков энергии в формате `A-B:C.D.E*F`. В шаблонах wb-mqtt-serial приводится в поле `description` канала.

| --- | --- | --- |
| `1-0:1.8.0` | Активная энергия (+), суммарная | `ap_energy_total` |
| `1-0:2.8.0` | Активная энергия (−), суммарная | `an_energy_total` |
| `1-0:3.8.0` | Реактивная энергия (+), суммарная | `rp_energy_total` |
| `1-0:4.8.0` | Реактивная энергия (−), суммарная | `rn_energy_total` |
| `1-0:21.7.0` | Активная мощность фазы 1 | `p_l1` |
| `1-0:41.7.0` | Активная мощность фазы 2 | `p_l2` |
| `1-0:61.7.0` | Активная мощность фазы 3 | `p_l3` |
| `1-0:32.7.0` | Напряжение фазы 1 | `urms_l1` |
| `1-0:52.7.0` | Напряжение фазы 2 | `urms_l2` |
| `1-0:72.7.0` | Напряжение фазы 3 | `urms_l3` |
| `1-0:31.7.0` | Ток фазы 1 | `irms_l1` |
| `1-0:51.7.0` | Ток фазы 2 | `irms_l2` |
| `1-0:71.7.0` | Ток фазы 3 | `irms_l3` |

## Приложение Е. Типовые шаблоны аварий

| Канал | EN (`title`) | RU | Назначение |

| `overcurrent_alarm` | Overcurrent Alarm | Перегрузка по току | Превышение допустимого тока |
| `overheating_alarm` | Overheating Alarm | Перегрев | Превышение допустимой температуры |
| `motion_alarm` | Motion Alarm | Тревога движения | Сработка датчика движения |
| `tamper_alarm` | Tamper Alarm | Попытка вмешательства | Срабатывание антисаботажа |
