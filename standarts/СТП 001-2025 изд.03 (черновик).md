# СТП 001-2025 «Шаблоны устройств для драйвера wb-mqtt-serial»

**Редакция 03. Черновик. Дата: 2026-05-25.**

> Настоящая редакция замещает изд.01 и изд.02 (внутренние черновики).
>
> Документ содержит только правила, специфичные для шаблонов wb-mqtt-serial:
>

> - правила оформления UI-текстов и переводов — в [СТП 003-2026](СТП%20003-2026%20\(UI-тексты,%20черновик\).md);

## 1. Область применения

Стандарт устанавливает правила формирования шаблонов устройств для драйвера `wb-mqtt-serial` Wiren Board. Шаблон — JSON-файл, описывающий устройство, его параметры настройки и контролы, публикуемые драйвером в MQTT.

Стандарт применяется ко всем новым шаблонам устройств Wiren Board. К партнёрским и пользовательским шаблонам — рекомендательно.

Существующие шаблоны мигрируются постепенно по отдельному плану миграции.

---

## 2. Принципы

Шаблон должен быть:

- **однозначно интерпретируемым** драйвером, валидатором и линтером;
- **понятным конечному пользователю** в веб-конфигураторе (вместе со стилем по СТП 003-2026);
- **миграционно совместимым** — при выпуске новой версии устройства имена сохраняемых сущностей не меняются (см. §10).

---

## 3. Структура шаблона

Шаблон — JSON-документ, соответствующий JSON-схеме `wb-mqtt-serial-device-template.schema.json`. Схема определяет обязательные и допустимые поля, типы значений и базовую структуру.

Настоящий стандарт устанавливает требования сверх схемы — содержательные правила формирования значений, канонические имена и стиль.

**Верхний уровень шаблона:**

```jsonc
{
    "device_type": "...",       // обязательно, см. §4
    "title": "...",             // отображаемое имя модели в каталоге устройств
    "group": "...",             // идентификатор группы устройств в каталоге (см. groups.json)
    "hw": [ ... ],              // соответствие физическому устройству
    "device": {
        "name": "...",          // отображаемое имя в UI (см. §4.3)
        "id": "...",            // идентификатор устройства, см. СТП 002-2026
        "translations": { ... }, // переводы (см. СТП 003-2026)
        "groups": [ ... ],      // см. §5
        "parameters": [ ... ],  // см. §6
        "channels": [ ... ],    // см. §7
        "setup": [ ... ]        // опционально, см. §9.6
    }
}
```

Поля `device.setup`, `device.translations`, `hw` — опциональны.

## 4. Идентификаторы, стиль текстов и канонические имена

### 4.1 Идентификаторы

Правила формирования идентификаторов (имён контролов, параметров, групп, `device.id`) определяются СТП 002-2026 «Идентификаторы MQTT-топиков в экосистеме Wiren Board».

Кратко: алфавит `[a-z0-9_]`, snake_case, не более четырёх слов и цифр, не более 64 символов, регулярное выражение `^[a-z][a-z0-9]*(_[a-z0-9]+)*$`.

### 4.2 Стиль текстов

Правила оформления текстов, видимых пользователю (`title`, `description`, значения `enum_titles`), определяются СТП 003-2026 «Стиль текстов в пользовательском интерфейсе Wiren Board».

Кратко: ASCII-кавычки и дефис; «е» вместо «ё»; точка как десятичный разделитель; EN — Title Case, RU — Sentence case; аббревиатуры заглавными; единицы в круглых скобках после названия и не в id; описания — короткие, без точки в конце.

### 4.3 Двуязычность и переводы

Многоязычные тексты (имена устройства, групп, контролов, параметров; описания; значения `enum_titles`) реализуются через секцию `device.translations`:

```jsonc
"device": {
    "translations": {
        "en": {
            "<key>": "English string",
            ...
        },
        "ru": {
            "<key>": "Русская строка",
            ...
        }
    }
}
```

Допустимы две модели формирования ключей; они могут применяться **в одном шаблоне** для разных текстов.

**Модель «английская-строка-как-ключ» — для коротких самодостаточных слов.**

Применяется, когда отображаемый английский текст короток (1-2 слова) и сам по себе является осмысленным идентификатором без дополнительной интерпретации: `General`, `Inputs`, `Outputs`, `Mode`, `Status`, `Debug`.

- В `name`/`title`/`description` указывается **английская отображаемая строка непосредственно** (в Title Case по СТП 003).
- Тот же английский текст служит ключом в `translations.ru` для русского перевода.
- Секция `translations.en` для этого ключа не нужна — драйвер использует ключ как английский текст.

Пример:

```jsonc
"groups": [
    {
        "id": "g_general",
        "title": "General"
    }
],
"translations": {
    "ru": {
        "General": "Общее"
    }
}
```

**Модель «идентификатор-как-ключ» — для длинных, составных или контекст-зависимых текстов.**

Применяется, когда отображаемый текст длинный (≥3 слов), технический, либо может различаться в разных частях шаблона: имена устройств, длинные подписи, описания параметров и каналов, тексты `enum_titles`.

- В `name`/`title`/`description` указывается **идентификатор** (snake_case или иной короткий машинный ключ).
- Тот же идентификатор используется как ключ в `translations.en` и `translations.ru`.
- Идентификатор пользователю не показывается; обе языковые версии задаются явно.

Пример:

```jsonc
"device": {
    "name": "wb_mr6cv3",
    "translations": {
        "en": {
            "wb_mr6cv3": "WB-MR6C v.3 (6-channel relay module)"
        },
        "ru": {
            "wb_mr6cv3": "WB-MR6C v.3 (6-канальный модуль реле)"
        }
    }
}
```

**Когда использовать какую модель.**

| Тип текста | Модель |
| --- | --- |
| Короткие группы (`General`, `Inputs`, `Debug`, `Hardware`) | английская-строка-как-ключ |
| Имя устройства с обозначением модели и пояснением | идентификатор-как-ключ |
| `title` параметра или `enum_titles` ≥ 3 слов | идентификатор-как-ключ |
| `description` параметра/канала | идентификатор-как-ключ |
| `name` канала (например, `Input 1`, `K 1`) | английская-строка-как-ключ |

Граница «короткое vs длинное» — практическая, не жёсткая; ориентир — 1-2 слова и однозначность смысла без контекста.

### 4.4 Поля устройства

Идентификация и отображение устройства в шаблоне задаются четырьмя полями.

| Поле | Описание и требования | Пример |
| --- | --- | --- |
| `device_type` | Машинный идентификатор семейства шаблона; привязка пользовательской конфигурации (`wb-mqtt-serial.conf`) к шаблону.<br>Должен совпадать с именем файла без префикса `config-`, суффикса `-deprecated` и расширения. Для шаблона с привязкой к версии прошивки — суффикс `-fw<N.M>` (§10.3). | `wb-mr6c-v3`<br>`wb-led-fw3.8` |
| `title` | Отображаемое имя шаблона в каталоге устройств веб-конфигуратора. Обычно длиннее `device.name`: содержит модель и краткое пояснение функционала. Ключ-идентификатор; отображаемые строки задаются в `device.translations` (§4.3) для обоих языков. | Ключ: `wb_mr6cv3_template_title`<br>EN: `WB-MR6C v.3 (6-channel relay with internal power supply)`<br>RU: `WB-MR6C v.3 (6-канальный модуль реле с внутренним блоком питания)` |
| `device.name` | Ключ отображаемого имени устройства в UI и в метаключе `/devices/.../meta/name`. Должно совпадать с маркировкой устройства, включая регистр и пунктуацию. Перевод (если нужен) — через `device.translations` (§4.3). | `WB-MR6C v.3` |
| `device.id` | Идентификатор устройства; драйвер формирует из него MQTT-топики `/devices/<device.id>/...`. Для новых шаблонов формируется по правилам СТП 002-2026 (snake_case, `[a-z0-9_]`). Для существующих шаблонов остается без изменений. После попадания шаблона в `master` не меняется (§10.2). | `wb_mr6c_v3` |

### 4.5 Канонические имена сущностей

Идентификаторы и переводы для сущностей, многократно встречающихся в WB-шаблонах (служебные каналы, стандартные группы, входы/выходы, счётчики, трёхфазные величины), определяются Приложениями А–Г настоящего стандарта.

Если сущность присутствует в приложениях — следует использовать каноническую форму, а не свою.

## 5. Группы (`groups[]`)

Группы определяют иерархию отображения параметров и каналов в UI веб-конфигуратора.

### 5.1 Иерархия и префиксы

Допускается до трёх уровней вложенности. Префикс идентификатора группы маркирует уровень:

| Уровень | Префикс | Пример |
| --- | --- | --- |
| Корневой | `g_` | `g_general`, `g_hw_info`, `g_in1` |
| Второй | `gg_` | `gg_in1_channels`, `gg_in1_press_params` |
| Третий | `ggg_` | `ggg_in1_out1_button_actions` |

### 5.2 Поля группы

```jsonc
{
    "id": "g_general",
    "title": "General",
    "group": null,                            // "<parent_group_id>"; или отсутствует для корневой группы
    "ui_options": {
        "wb": {
            "disable_title": false
        }
    }
}
```

Поле `title` — отображаемое имя группы в UI (по правилам СТП 003-2026).

### 5.3 Канонические корневые группы

См. Приложение Б. При наличии соответствующей сущности её следует помещать в каноническую группу.

## 6. Параметры (`parameters[]`)

Параметры — поля настройки устройства, выставляемые пользователем в веб-конфигураторе при создании конфигурации (записываются в устройство через `setup` или при назначении).

### 6.1 Основные поля

```jsonc
{
    "id": "baud_rate",
    "title": "Baud Rate",
    "description": "...",                       // опционально, по правилам СТП 003-2026
    "address": "...",                           // см. §8
    "reg_type": "holding",                      // см. §8
    "format": "u16",                            // см. §8
    "default": 9600,
    "enum": [96, 192, 384, 576, 1152],
    "enum_titles": ["9600", "19200", ...],
    "group": "g_general",
    "required": true,                           // true или false
    "condition": "...",                         // см. §9.3
    "fw": "1.23.0"                              // см. §9.4 (минимальная версия)
}
```

### 6.2 Идентификаторы параметров

Для `parameters[].id` допустимы три формы:

| Форма | Когда применять | Пример |
| --- | --- | --- |
| **Описательный snake_case** | Технические/коммуникационные параметры (скорость обмена, таймауты, общие режимы) и параметры устройств, где смысл выражается осмысленным словом | `baud_rate`, `safety_timer_s`, `disable_indication` |
| **`p<N>` — позиционный** | Устройства с десятками параметров без естественных имён и без панели с кодами (некоторые контроллеры, реле в общем стиле производителя) | `p1`, `p2`, …, `p47` |
| **Panel-код** | Устройства с этикеточными кодами параметров (частотные преобразователи, ПЛК), где код — общеизвестная сущность из документации | `f0_04`, `pn_01_03`, `a01` |

**Правило выбора формы в шаблоне:**

- **Описательная форма** допускается всегда — для технических/коммуникационных параметров (`baud_rate`, `slave_id_offset`, таймауты, общие режимы), для которых описательное имя естественно. Эти параметры присутствуют в шаблоне в описательной форме независимо от того, какая форма выбрана для основной массы.
- **Для основной массы device-specific параметров** (специфика конкретного семейства устройств — режимы работы, настройки функционала) выбирается **одна** форма: либо `p<N>`, либо panel-код. Смешивать `p<N>` и panel-код в одном шаблоне для основной массы не допускается.

Пример комбинации в шаблоне частотного преобразователя:

| id | Форма | Назначение |
| --- | --- | --- |
| `baud_rate` | описательная | скорость обмена RS-485 |
| `safety_timer_s` | описательная | таймер безопасного режима |
| `f0_04` | panel-код | параметр панели F0.04 (основная масса) |
| `f0_05` | panel-код | параметр панели F0.05 |
| `pn_01_03` | panel-код | параметр панели Pn-01.03 |

### 6.3 Нормализация panel-кодов

Panel-код устройства часто содержит точку или дефис (например, `F0.04`, `Pn-01.03`). В id параметра он приводится к нормализованной форме:

- символы `.`, `-`, `:` заменяются на `_`;
- буквы приводятся к нижнему регистру;
- ведущие нули не убираются.

Примеры:

| Panel-код на устройстве | id в шаблоне |
| --- | --- |
| `F0.04` | `f0_04` |
| `Pn-01.03` | `pn_01_03` |
| `A01` | `a01` |

В `title`/`description` panel-код сохраняется в исходной маркировке: `title: "F0.04 - Frequency setting"`.

### 6.4 Типы значений

| Тип параметра | Признак |
| --- | --- |
| Целое число | `format: u8 / s8 / u16 / s16 / u32 / s32 / u64 / s64` |
| Число с плавающей точкой | `format: float / double` |
| Перечислимое | поле `enum`, опционально `enum_titles` |
| Логическое (двоичное) | `format: u8` + `enum: [0, 1]` + `enum_titles` |
| BCD | `format: bcd16 / bcd32` |
| Строка | `format: string` + `string_data_size` |

### 6.5 Группа параметра

`parameters[].group` — id группы (см. §5), в которой параметр отображается. Если не указано — корень.

## 7. Каналы (`channels[]`)

Каналы — контролы устройства, публикуемые драйвером в MQTT и отображаемые пользователю в реальном времени.

### 7.1 Поля `id` и `name`

В канале действуют **два разных** поля для именования:

| Поле | Назначение | Правила |
| --- | --- | --- |
| `id` | Имя MQTT-контрола; топик публикуется как `/devices/<device.id>/controls/<id>` | snake_case, см. СТП 002-2026 |
| `name` | Отображаемое имя канала (в UI, в диагностике); используется как ключ для `device.translations`. Используется в `wb-mqtt-serial.conf`. | Title Case в EN, см. СТП 003-2026 |

Если `id` не задан, драйвер использует значение `name` для формирования MQTT-топика — это поведение **сохраняется только для legacy-шаблонов**. В новых шаблонах поле `id` указывается явно для каждого канала.

### 7.2 Основные поля канала

```jsonc
{
    "name": "Input 1",                  // отображаемое имя (СТП 003-2026)
    "id": "input_1",                    // имя MQTT-топика (СТП 002-2026)
    "type": "switch",                   // тип контрола, см. §7.3
    "reg_type": "discrete",             // см. §8
    "address": 12,                      // см. §8
    "format": "u16",                    // см. §8
    "scale": 0.01,                      // опционально
    "offset": 0,                        // опционально
    "units": "V",                       // см. СТП 003-2026 §5
    "min": 0,
    "max": 250,
    "round_to": 0.01,
    "readonly": true,                   // true или false
    "sporadic": true,                   // см. §7.4
    "enabled": false,                   // канал по умолчанию выключен
    "error_value": 65535,
    "group": "gg_in1_channels",
    "condition": "...",                 // см. §9.3
    "fw": "1.20.0",                     // см. §9.4 (минимальная версия)
    "write_address": 100                // см. §9.5
}
```

### 7.3 Типы контролов

Типы — из конвенции [Wiren Board MQTT Conventions §3](https://github.com/wirenboard/conventions#3-control-types) (значения метаключа `meta/type`).

**Функциональные типы.** Применяются по назначению:

| Тип | Смысл |
| --- | --- |
| `value` | Числовое значение общего назначения. **Значение по умолчанию**; поле `type` для таких каналов опускается. Применяется в сочетании с `units` для физических величин |
| `switch` | Двухпозиционный переключатель (0/1) |
| `pushbutton` | Нажимная кнопка (без состояния) |
| `range` | Целочисленный диапазон с `min`/`max` |
| `text` | Произвольный текст |
| `rgb` | Цвет в формате `R;G;B` |
| `wo-switch` | Переключатель только для записи |
| `alarm` | Двоичный индикатор тревоги (0/1) |
| `unixtime` | Время в формате Unix-времени |
| `w1-id` | Идентификатор Dallas 1-Wire |

**Специфические физические типы (deprecated).** В схеме `wb-mqtt-serial-confed-common.schema.json` явно помечены как устаревшие (через суффикс `(deprecated, use value with units)` в `enum_titles`): `temperature`, `rel_humidity`, `atmospheric_pressure`, `rainfall`, `wind_speed`, `power`, `power_consumption`, `voltage`, `water_flow`, `water_consumption`, `resistance`, `concentration`, `heat_energy`, `heat_power`, `dimmer`, `lux`, `pressure`, `current`, `sound_level`. В новых шаблонах применять не следует.

Список валидных значений `type` ограничен этим перечнем (функциональные + deprecated). Любые иные значения (например, `energy_power`) — невалидны.

**Правило для новых шаблонов.** Для физических величин поле `type` опускается (подразумевается `value`); единица измерения задаётся полем `units` по словарю единиц конвенции (Wiren Board MQTT Conventions §3, подраздел «Units»).

Пример канала температуры (канонический служебный канал, см. Приложение А):

```jsonc
{
    "name": "MCU Temperature",
    "id": "mcu_temperature",
    "reg_type": "input",
    "address": 124,
    "format": "s16",
    "scale": 0.1,
    "units": "deg C",
    "group": "g_hw_info"
}
```

(поле `type` опущено — подразумевается `value`).

### 7.4 Спорадические каналы

Поле `sporadic: true` указывает драйверу использовать Быстрый Modbus для опроса канала; значение публикуется только при наличии события на стороне устройства. Требуется поддержка со стороны прошивки.

### 7.5 Группа канала

`channels[].group` — id группы (см. §5), в которой канал отображается.

## 8. Адресация и форматы значений

### 8.1 `reg_type`

Допустимые значения определяются JSON-схемой. Типичные:

| Значение | Назначение |
| --- | --- |
| `holding` | Modbus holding registers (читать/писать) |
| `input` | Modbus input registers (только чтение) |
| `coil` | Modbus coils (битовые, читать/писать) |
| `discrete` | Modbus discrete inputs (битовые, только чтение) |
| `press_counter` | Регистр счётчика нажатий (WB-специфика) |

### 8.2 `address`

Адрес регистра, число или строка. Поддерживаемые формы:

- целое число: `123`;
- шестнадцатеричное в строке: `"0x7B"`;
- битовое: `"<reg>:<bit_offset>:<bit_width>"` (см. §9.1).

### 8.3 `format`

| Значение | Размер регистров |
| --- | --- |
| `u8`, `s8` | 1 |
| `u16`, `s16` | 1 |
| `u32`, `s32`, `float`, `bcd32` | 2 |
| `u64`, `s64`, `double` | 4 |
| `string` | определяется `string_data_size` |

### 8.4 `word_order`

Для многорегистровых форматов: `big_endian` (по умолчанию, MSB first) или `little_endian` (LSB first). Уточняется по даташиту устройства.

### 8.5 `scale` и `offset`

Преобразование сырого значения регистра в физическую величину:

```
displayed = raw * scale + offset
```

`scale` — обычно степень десяти (`0.1`, `0.01`, `0.001`), реже произвольное.

## 9. Специальные случаи

### 9.1 Битовые поля

Битовый канал/параметр имеет адрес в формате `"<reg>:<bit_offset>:<bit_width>"` (три компонента, разделённые двоеточиями). Один бит задаётся шириной `1`.

```jsonc
{
    "name": "Alarm 1",
    "id": "alarm_1",
    "reg_type": "input",
    "address": "0x100:5:1",
    "type": "alarm"
},
{
    "name": "Error",
    "id": "error",
    "reg_type": "input",
    "address": "0x100:0:4"
}
```

Bit offset считается от младшего бита. Поле не должно пересекать границы регистра.

Регулярное выражение схемы для битового адреса: `^(?:0x[A-Fa-f0-9]+|\d+)(?::\d+:\d+)?$` — компоненты `offset` и `width` либо присутствуют оба, либо отсутствуют оба; форма `"<reg>:<bit_offset>"` без `<bit_width>` недопустима.

### 9.2 Многорегистровые параметры

Для `format` размером > 1 регистра поле `address` указывает первый регистр. Порядок слов определяется `word_order`.

```jsonc
{
    "name": "Energy Total",
    "id": "energy_total",
    "reg_type": "input",
    "address": 1024,
    "format": "u32",
    "word_order": "big_endian",
    "units": "kWh"
}
```

(поле `type` опущено — подразумевается `value`; единицы измерения — по словарю Wiren Board MQTT Conventions §3).

### 9.3 Условия (`condition`)

Поле `condition` — логическое выражение, ссылающееся на id параметров. Если выражение возвращает `false`, элемент не активен (скрывается, не опрашивается, не записывается).

```jsonc
{
    "id": "input_1",
    "name": "Input 1",
    "reg_type": "discrete",
    "address": 0,
    "type": "switch",
    "group": "gg_input_1_counters",
    "condition": "encoder_1_mode==0",
    "sporadic": true
},
```

**Грамматика.** Поддерживается **ограниченное JS-подобное выражение** (не произвольный JavaScript). Парсер драйвера (`src/expression_evaluator.{h,cpp}`) принимает:

- сравнения: `==`, `!=`, `>`, `<`, `>=`, `<=`;
- логические операторы: `&&`, `||`;
- скобки для группировки: `(`, `)`;
- целочисленные литералы (с возможным ведущим `-`);
- идентификаторы параметров (snake_case);
- единственную функцию `isDefined(<id>)` — `true`, если параметр с указанным `id` определён в текущей конфигурации.

Любые иные конструкции JS — унарное отрицание `!`, арифметика, строки, тернарный оператор, прочие функции — **не поддерживаются**.

Идентификаторы в выражении — id параметров (не каналов и не топиков); должны существовать в `parameters[]` шаблона.

Пробелы в выражении не допускаются (лексер не пропускает пробельных символов).

Примеры:

```jsonc
"condition": "encoder_1_mode==0"
"condition": "isDefined(safety_mode_source)&&safety_mode_source!=1"
"condition": "(mode==1)||(mode==3)"
```

### 9.4 Зависимость от прошивки (`fw`)

Поле `fw` — строка с **минимальной** версией прошивки, начиная с которой элемент поддерживается. Указывается как чистая версия (`"1.2.3"`), без операторов сравнения и без диапазонов. Если прошивка устройства ниже — элемент не активен. Отсутствие поля означает поддержку любой версии прошивки.

```jsonc
{
    "name": "MCU Voltage",
    "id": "mcu_voltage",
    "fw": "1.20.0",
    ...
}
```

Если в `parameters[]` присутствуют несколько записей с одним и тем же `id` (например, разные варианты под `condition`), значение `fw` у них должно совпадать (требование README `wb-mqtt-serial`).

### 9.5 Разделённый адрес записи (`write_address`)

Применяется, когда регистр имеет разные адреса для чтения и записи. Также используется для нестандартных бинарных значений в паре с `on_value` / `off_value`:

```jsonc
{
    "name": "K 1",
    "id": "k1",
    "reg_type": "holding",
    "address": 100,
    "write_address": 200,
    "type": "switch"
}
```

```jsonc
{
    "name": "Door Lock",
    "id": "door_lock",
    "reg_type": "holding",
    "address": 50,
    "on_value": "0x00a5",
    "off_value": "0x005a",
    "type": "switch"
}
```

### 9.6 Setup-операции (`setup[]`)

Массив операций, выполняемых драйвером при подключении устройства (запись начальных значений в регистры). Используется для устройств, требующих инициализации.

**Рекомендация.** Предпочтительный способ задания настроек устройства — `parameters[]` (§6): значение выставляется пользователем явно в веб-конфигураторе и хранится в его конфигурации. `setup[]` применяется как крайний случай — когда регистр обязательно нужно проинициализировать при подключении устройства, но не нужно отображать в интерфейсе.

В элементе `setup` поддерживается поле `condition` (см. §9.3) — операция выполняется только если выражение истинно (т. е. зависит от значений `parameters[]`). Поле `fw` в `setup` не поддерживается — операции выполняются всегда, независимо от версии прошивки.

```jsonc
"setup": [
    {
        "title": "Set Modbus delay",
        "address": 200,
        "reg_type": "holding",
        "value": 5
    },
    {
        "title": "Enable curtain mode",
        "address": 1280,
        "reg_type": "holding",
        "value": 1,
        "condition": "curtain1_mode==1"
    }
]
```

## 10. Имя файла и версионирование шаблонов

### 10.1 Имя файла

Имя файла шаблона строится из `device_type` с добавлением префикса `config-` и расширения `.json` или `.json.jinja`. Например:

- `device_type: "wb-mr6c-v3"` → `config-wb-mr6c-v3.json.jinja`
- `device_type: "wb-led-fw3.8"` → `config-wb-led-fw3.8.json.jinja`

Расширение `.json` используется для статических JSON-шаблонов, `.json.jinja` — для шаблонов, рендеримых через Jinja2.

Для шаблонов, привязанных к конкретной версии прошивки, и для deprecated-шаблонов имя файла получает дополнительные суффиксы — см. §10.3 и §10.4.

### 10.2 Обратная совместимость

При выпуске новой ревизии устройства, не нарушающей обратной совместимости, изменения вносятся в существующий шаблон.

**Что нельзя изменять после попадания шаблона в `master`:**

- `id` каналов и `id` параметров — изменение или удаление запрещено; допускается только добавление новых;
- `device.id` — не меняется (его наличие фиксирует имя MQTT-устройства);
- `device_type` — не меняется (сохраняется в `wb-mqtt-serial.conf` для привязки к шаблону).

**Что можно изменять:**

- имя файла шаблона;
- `name`/`title`/`description` каналов, групп, параметров;
- `device.name`;
- содержимое `device.translations`;
- `group` каналов и параметров (логическая группировка в UI).

Эти поля влияют только на отображение, не на топики и не на пользовательские конфиги.

**Добавление новых элементов в существующий шаблон.**

Допускается:
- добавление нового канала или параметра;
- добавление нового варианта параметра (новой строки в `parameters[]` с тем же `id`, но с другим `condition`) — без изменения существующих `condition`;

При любом таком добавлении:
- у нового элемента **обязательно** указывается поле `fw` — минимальная версия прошивки, с которой элемент доступен (§9.4);
- новый канал по умолчанию имеет `"enabled": false` (пользователь активирует его сам).

**Запрещено в рамках совместимой эволюции:** изменение существующих `condition`, изменение существующих значений `enum`, удаление каналов или параметров. Для таких изменений см. §10.3.

### 10.3 Шаблон для конкретной версии прошивки

Применяется, когда выпускается новая ревизия устройства с прошивкой, в которой регистровая карта или поведение существенно изменились, и часть шаблона должна работать **только** на этой и более поздних прошивках.

**Когда правим существующий шаблон, а когда создаём новый:**

| Изменение | Куда |
| --- | --- |
| Добавить новый канал или параметр | в существующий шаблон (с обязательным `fw`, см. §10.2) |
| Добавить новый вариант параметра (новая строка с тем же `id`, новое `condition`), не меняя существующих | в существующий шаблон (с `fw`) |
| Изменить существующее значение `enum` | новый шаблон под новую прошивку |
| Изменить существующий `condition` | новый шаблон под новую прошивку |
| Изменить адрес, тип, формат существующего элемента | новый шаблон под новую прошивку |
| Удалить канал или параметр | новый шаблон под новую прошивку |

Все строки «новый шаблон под новую прошивку» подразумевают, что изменение обусловлено выпуском новой версии прошивки устройства. Если требуется изменить шаблон **без** изменения версии прошивки (например, исправить ошибку в самом шаблоне) — применяется отдельный путь с суффиксом `-tpl<N>` (см. §10.4); это редкая операция.

**Признак применения настоящего раздела:** для всего шаблона задана минимальная версия прошивки в блоке `hw[].fw` (т. е. шаблон применим только к устройствам с прошивкой не ниже указанной).

**Соглашение:**

- к `device_type` добавляется суффикс `-fw<N.M>`, где `<N.M>` — минимальная версия прошивки;
- имя файла шаблона **точно соответствует** значению `device_type` (см. §4.4);
- в `hw[].fw` указывается фактическая минимальная версия прошивки.

**Пример** (файл `config-wb-led-fw3.8.json.jinja`):

```jsonc
{
    "device_type": "wb-led-fw3.8",
    "hw": [
        {
            "signature": "WB-LED",
            "fw": "3.8.0"
        }
    ],
    ...
}
```

**Покрытие версий прошивок.** Для каждой версии прошивки устройства, с которой оно выпускалось, должен существовать ровно один **валидный (не помеченный `deprecated`) шаблон**, подходящий по сигнатуре и `hw[].fw`. При сканировании устройств выбирается шаблон с максимальным `hw[].fw`, не превышающим фактическую версию прошивки устройства.

При выпуске шаблона для новой прошивки старый шаблон (для устройств с более ранней прошивкой) `deprecated` **не помечается** — иначе сломается поиск шаблона при сканировании устройств со старой прошивкой. Старый шаблон остаётся валидным, его задача — обслуживать устройства с прошивкой ниже `<N.M>`.

Несколько прошивочных ревизий могут сосуществовать:

- `config-wb-led.json.jinja` с `device_type: "wb-led"` — для устройств без `fw`-привязки;
- `config-wb-led-fw3.8.json.jinja` с `device_type: "wb-led-fw3.8"`;
- `config-wb-led-fw4.0.json.jinja` с `device_type: "wb-led-fw4.0"`.

Все они валидны; драйвер выбирает подходящий по сигнатуре устройства и фактической версии его прошивки.

### 10.4 Маркер deprecated

Шаблон помечается как `deprecated`, когда его нужно убрать из списка выбора в каталоге веб-конфигуратора и из поиска при сканировании устройств:

- в имя файла добавляется суффикс `-deprecated`: `config-<device>-deprecated.json[.jinja]`;
- в верхнем уровне шаблона устанавливается `"deprecated": true`.

Драйвер продолжает обрабатывать **существующие конфигурации**, использующие deprecated-шаблон, но новые устройства через него не добавляются.

**Когда помечать deprecated:**

- шаблон полностью заменён более новым с совместимыми именами топиков (миграция конкретного устройства).

**Когда НЕ помечать deprecated:**

- при выпуске нового шаблона для новой версии прошивки старый шаблон **остаётся валидным** для устройств с более ранней прошивкой (см. §10.3).

Маркер `deprecated` совместим с прошивочным суффиксом `-fw<N.M>` — например, `config-wbe2-i-opentherm-fw1.3-deprecated.json` для устаревшего шаблона прошивки 1.3.

#### Замена шаблона без изменения версии прошивки (`-tpl<N>`)

Редкая операция. Применяется, когда нужно внести обратно-несовместимые изменения в шаблон, **не связанные** с выпуском новой прошивки устройства (например, исправление ошибки в самом шаблоне). Привязка к версии прошивки при этом не меняется.

Для исходного шаблона `device_type` сохраняется без изменений (чтобы не сломать существующие конфигурации); в имя файла добавляется суффикс `-deprecated`, в верхнем уровне шаблона устанавливается `"deprecated": true`.

Для нового шаблона (заменяющего deprecated) в `device_type` и в имя файла добавляется суффикс `-tpl1`/`-tpl2` и т. д.

Пример:

| Сущность | Изменение в существующем шаблоне | Значение для нового шаблона |
| --- | --- | --- |
| `device_type` | `"wb-mr6c-v3"` (остаётся без изменений) | `"wb-mr6c-v3-tpl1"` |
| Имя файла | меняется на `config-wb-mr6c-v3-deprecated.json.jinja` | `config-wb-mr6c-v3-tpl1.json.jinja` |

## Приложение А. Канонические служебные каналы

Служебные каналы — каналы устройств Wiren Board, общие для большинства семейств; размещаются в группе `g_hw_info` (см. Приложение Б).

| `id` | `name` (EN) | RU перевод | Назначение |
| --- | --- | --- | --- |

| `uptime` | Uptime | Время работы с момента включения | Время с включения, секунд |
| `hw_batch_number` | HW Batch Number | Номер партии | Номер партии аппаратуры |
| `mcu_temperature` | MCU Temperature | Температура МК | Температура микроконтроллера |
| `mcu_voltage` | MCU Voltage | Напряжение питания МК | Напряжение питания микроконтроллера |
| `supply_voltage` | Supply Voltage | Напряжение питания | Напряжение питания устройства |
| `min_supply_voltage_since_startup` | Min Supply Voltage Since Startup | Мин. напряжение питания с момента включения | Минимум за время работы |

**Примечания:**

- Тип контролов — `value` (поле `type` опускается, см. §7.2). Специфические типы (`temperature`, `voltage`, …) deprecated по конвенции Wiren Board MQTT Conventions и в новых шаблонах не применяются.
- Единицы измерения (вольты, секунды) указываются в метаключе `units`, а не в `title` или `name` (см. СТП 003-2026 §5).
- Все служебные каналы по умолчанию имеют `enabled: false` (включаются пользователем при необходимости).

| `gg_in<N>_out<M>_button_actions` | Действия кнопки для конкретного выхода |

## Приложение В. Канонические каналы цифровых входов и выходов

### В.1 Дискретные входы

| `id` | `name` (EN) | RU перевод | Назначение |
| --- | --- | --- | --- |
| `input_<N>` | Input N | Вход N | Текущее состояние дискретного входа N |

`<N>` — номер входа (`input_1`, `input_2`, …).

### В.2 Релейные выходы

| `id` | `name` (EN) | RU перевод | Назначение |
| --- | --- | --- | --- |
| `k<N>` | K N | K N | Релейный выход N (по этикетке на устройстве) |
| `relay_<N>` | Relay N | Реле N | Альтернатива для устройств без буквенной маркировки |

Для устройств Wiren Board с маркировкой `K1`, `K2`, … применяется `k<N>`.

### В.3 Счётчики нажатий

Для каждого входа `<N>`, поддерживающего распознавание нажатий, определяются следующие счётчики:

| `id` | `name` (EN) | RU перевод |
| --- | --- | --- |
| `input_<N>_counter` | Input N Counter | Вход N: счетчик замыканий |
| `input_<N>_single_press_counter` | Input N Single Press Counter | Вход N: счетчик одиночных нажатий |
| `input_<N>_long_press_counter` | Input N Long Press Counter | Вход N: счетчик длинных нажатий |
| `input_<N>_double_press_counter` | Input N Double Press Counter | Вход N: счетчик двойных нажатий |
| `input_<N>_shortlong_press_counter` | Input N Shortlong Press Counter | Вход N: счетчик коротких, а затем длинных нажатий |
| `reset_all_counters` | Reset All Counters | Сбросить все счетчики |

**Примечания:**

- Употребляется «Single Press», не «Short Press» (по противопоставлению с «Long» и «Double»).
- «Shortlong» пишется слитно (как в id, так и в `title`).
- Все счётчики имеют `reg_type: "press_counter"`, `sporadic: true`.

## Приложение Г. Канонические трёхфазные электрические величины

Применяются для счётчиков энергии и измерителей мощности.

### Г.1 Однофазные/трёхфазные счётчики

| `id` | `name` (EN) | RU перевод | Единицы (`units`) |
| --- | --- | --- | --- |
| `frequency` | Frequency | Частота сети | Hz |
| `urms_l<N>` | Urms L*N* | Напряжение фазы *N* | V |
| `irms_l<N>` | Irms L*N* | Ток фазы *N* | A |
| `p_l<N>` | P L*N* | Активная мощность фазы *N* | W |
| `q_l<N>` | Q L*N* | Реактивная мощность фазы *N* | var |
| `s_l<N>` | S L*N* | Полная мощность фазы *N* | VA |
| `pf_l<N>` | PF L*N* | Коэффициент мощности фазы *N* | — |
| `voltage_angle_l<N>` | Voltage Angle L*N* | Угол напряжения фазы *N* | ° |
| `ap_energy_l<N>` | AP Energy L*N* | Активная энергия (+) фазы *N* | kWh |
| `an_energy_l<N>` | AN Energy L*N* | Активная энергия (−) фазы *N* | kWh |
| `rp_energy_l<N>` | RP Energy L*N* | Реактивная энергия (+) фазы *N* | kvarh |
| `rn_energy_l<N>` | RN Energy L*N* | Реактивная энергия (−) фазы *N* | kvarh |
| `thdn_u_l<N>` | THDN U L*N* | THDN напряжения фазы *N* | % |
| `thdn_i_l<N>` | THDN I L*N* | THDN тока фазы *N* | % |

`<N>` — номер фазы (`1`, `2`, `3`).

### Г.2 Многоканальные счётчики (MAP-серия)

Для счётчиков, измеряющих несколько физических вводов (12-канальные и т. п.), используется двухуровневая нотация: `channel<K>_<величина>_l<N>`.

Примеры: `channel1_urms_l1`, `channel1_p_l2`, `channel4_ap_energy_l3`.

Группировка — в `channel<K>_l<N>` (фазная подгруппа канала) и `channel<K>_total` (итог по каналу).

### Г.3 Тип канала

Тип всех каналов настоящего приложения — `value` (поле `type` опускается, см. §7.2). Единицы измерения берутся из словаря единиц Wiren Board MQTT Conventions §3.
