# Открытые вопросы и упущения по сравнению с изд.01

Рабочий документ — список того, что осталось нерешённым или потерялось при переходе от изд.01 (монолитный 417-строчный черновик) к текущему пакету (СТП 001 изд.03 + СТП 002-2026 + СТП 003-2026).

**Не нормативный документ** — вход в очередь обсуждений. По итогам решений добавляется в СТП либо помечается «сознательно дропнуто».

Дата: 2026-06-15.

---

## 1. Упущения по сравнению с изд.01

Покрытие изд.01 в новой редакции — около 85% по смыслу. Большая часть «пропусков» — сознательная замена масок на канонические имена (Приложения А-Г СТП 001) и вынос общих правил в СТП 002/003. Ниже — 13 конкретных пунктов из изд.01, **отсутствующих** в новой редакции в явном нормативном виде.

| Тема изд.01 | Статус в новой редакции |
| --- | --- |
| Термины «настроечный регистр» / «оперативный регистр» | потеряны полностью; смысл косвенно через §6 (parameters) и §7 (channels), но определений нет |
| Маска `device_type` «[бренд]_[модель1]_[модель2]» | заменена примерами в §4.4 без обобщения |
| Маска `title` шаблона «[БРЕНД] [модель1], [модель2], …» | заменена общим «модель + краткое пояснение функционала» |
| Правило `{id}_description` для комментариев параметров | не воспроизведено |
| Правило `{p_id}_enum_{N}` для ключей `enum_titles` | не воспроизведено |
| Рекомендация `[значение]_[название]` в enum_title (`0_Normal`, `1_Search F`) для удобства наладки по документации | потеряно |
| Маски `alarm_*`, `warning_*`, `error_*` для идентификаторов аварий | заменены каноном из Прил. В СТП 001, но **только для типовых аварий** (overvoltage_alarm и т. п. — а они тоже убраны); общая маска для произвольных аварий не описана |
| Маски `set_*`, `cmd_*`, `state_*` для уставок/команд/статусов | не воспроизведены |
| Маска `v_<клемма>` для вычисляемых величин | не воспроизведена |
| Правило «общая часть в начале идентификатора, отличие — в конце» (`vbus_raw_charger` / `vbus_raw_discharger`) | не воспроизведено |
| Маски наименований аварий «АВАРИЯ/ALARM: <величина>», «ВНИМАНИЕ/WARNING: …», «ОШИБКА/ERROR: …» | потеряны |
| Маски «Уставка: …», «Команда: …» в `title` контрола | потеряны |
| Префикс «[Идентификатор производителя]» в `title` настроечного параметра (`F1.00: Предустановленная F`) | не нормирован, хотя §6.3 СТП 001 говорит о panel-кодах в `id` |

### Возможные категории

- **Сознательно дропнуто.** Например, маски аварий, заменённые жёстким каноном Прил. В.
- **Случайно потеряно.** Например, термины «настроечный/оперативный регистр» — стоит вернуть как определения.
- **Под вопросом.** Например, правила `{id}_description` и `{p_id}_enum_{N}` — полезные конвенции, но требуют согласования с фактической практикой шаблонов.

## 2. Открытые вопросы

### 2.1 Длина канонических имён > 4 слов

**Противоречие.** СТП 002-2026 §2.3 (со ссылкой на «Текст в веб-интерфейсе» и conventions §1.2) требует «не более четырёх смысловых единиц». Но в Приложениях А, В, Г СТП 001 есть имена, нарушающие правило:

| Идентификатор | Единиц |
| --- | --- |
| `min_supply_voltage_since_startup` | 5 |
| `min_mcu_voltage_since_startup` | 5 |
| `input_<N>_single_press_counter` | 5 (включая `<N>`) |
| `input_<N>_double_press_counter` | 5 |
| `input_<N>_shortlong_press_counter` | 5 |
| `channel<K>_ap_energy_l<N>` (Прил. Г.2) | 6 |

**Варианты решения:**

- **(а)** Смягчить §2.3 СТП 002: добавить оговорку «правило применяется к смысловому ядру; индексные плейсхолдеры (`<N>`, `<K>`), стандартные префиксы (`min_*`, `channel<K>_*`) не учитываются при подсчёте».
- **(б)** Зафиксировать в Приложениях А-Г СТП 001 явное исключение: «канонические имена настоящего стандарта могут превышать лимит §2.3 СТП 002».
- **(в)** Сократить канонические имена. Болезненно: ломает существующие шаблоны.

Предпочтение — **(а)** (смягчение правила), как наиболее аккуратное.

### 2.2 Возврат масок именования из изд.01

Часть масок из изд.01 (см. §1 выше) представляют **полезные конвенции**, утрату которых стоит обсудить.

**Кандидаты на возврат** (с пометкой, куда добавлять):

| Маска | Назначение | Куда |
| --- | --- | --- |
| `{id_параметра}_description` | Идентификатор комментария параметра | СТП 001 §6 — как часть «модели идентификатор-как-ключ» |
| `{p_id}_enum_{N}` | Ключ перевода для значения `enum_titles` | СТП 001 §6 — рядом с правилами enum |
| `alarm_<сигнал>`, `warning_<сигнал>`, `error_<сигнал>` | Идентификаторы аварий/предупреждений/ошибок | СТП 001 §7 или новый раздел «канонические маски» |
| `set_<величина>`, `cmd_<действие>`, `state_<сущность>` | Идентификаторы уставок / команд / статусов | СТП 001 §7 |
| `v_<клемма>` | Вычисляемая величина, привязанная к клемме | СТП 001 §7 |
| «общая часть в начале» (`vbus_raw_charger` / `_discharger`) | Группировка связанных каналов префиксом | СТП 001 §7.1 |
| Префикс производителя в `title` параметра (`F1.00: …`) | Сохранение panel-кода в title | СТП 001 §6.3 — рядом с нормализацией panel-кодов |

**Варианты решения:**

- **(а)** Вернуть все нормативно. Самый строгий канон, но требует проверки соответствия существующим шаблонам.
- **(б)** Вернуть как рекомендации (СЛЕДУЕТ, не ДОЛЖЕН). Гибче.
- **(в)** Вернуть только те, что подтверждены практикой текущих шаблонов (по анализу `templates/`).
- **(г)** Не возвращать — оставить в инвентаризации как историческую справку.

Для решения нужна оценка: насколько эти маски **уже используются** в существующих шаблонах. Можно поручить отдельному обзору `templates/*.json[.jinja]`.

### 2.3 Термины «настроечный регистр» / «оперативный регистр»

Утрачены при сужении области применения. Их полезно вернуть как короткое определение в §3 «Структура шаблона» СТП 001 — это терминология, которой пользуются авторы шаблонов и техподдержка.

**Решение:** скорее всего «вернуть как краткие определения», без отдельного раздела. Низкий приоритет.

### 2.4 Маска `device_type` и `title` шаблона

Изд.01 предписывал явные маски:

- `device_type` = `"<бренд>_<модель1>_<модель2>"` (например, `wirenboard_wb-mr3lv_wb-mr3hv_wb-mrwl3`);
- `title` = `"[БРЕНД] <модель1>, <модель2>, …"`.

Новая редакция показывает только примеры в §4.4 без обобщения.

**Решение:** скорее всего вернуть как **рекомендации** в §4.4. Низкий приоритет, влияет только на новые шаблоны для мультимодельных семейств.

## 3. Решённые вопросы (для истории)

Краткая запись того, что **обсуждалось и закрыто** в текущей итерации — чтобы не возвращаться:

- **«Другой синтаксис enum в каналах vs параметрах»** — на уровне шаблона разницы нет, синтаксис тот же (array `enum` + array `enum_titles`). Различие — только при публикации в MQTT meta (объектная форма по conventions §2.2). §7.6 в СТП 001 убран после короткого обсуждения.
- **`tpl<N>` vs `-fw<N.M>`** — два разных пути: `-fw<N.M>` для шаблона под новую прошивку, `-tpl<N>` для редкой замены deprecated без изменения версии. Зафиксировано в §10.3 и §10.4 СТП 001.
- **«Размер отступа в JSON-примерах»** — 4 пробела, по образцу `templates/config-*.json`.
- **«ё vs е» в самих СТП** — формально правило §3.1 СТП 003 регулирует UI-тексты, не нормативные документы. В таблицах внутри СТП, демонстрирующих UI-стиль (`§7 «Особое написание»`), применяется «е». В прозаическом тексте СТП допускается «ё».
