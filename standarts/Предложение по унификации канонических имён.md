# Предложение по унификации канонических имён каналов

**Статус:** черновик для обсуждения с командой. Не нормативный документ.
**Дата:** 2026-06-15.
**Контекст:** надстройка над СТП 001-2025 изд.03 (Приложения А, В, Г).

---

## Идея

Унифицировать имена контролов в шаблонах wb-mqtt-serial по схеме:

```
<префикс_величины>_[модификатор_]<сущность>
```

- **префикс величины** — обозначение измеряемой физической или дискретной величины;
- **модификатор** (опционально) — уточнение (`min`/`max`, направление энергии, вид нажатия и т. п.);
- **сущность** — что/где измеряется (`mcu`, `vin`, `l1`, `in1`, `ambient`, …).

Применяется **только к новым шаблонам**. Существующие шаблоны не мигрируются.

---

## Закрытый словарь префиксов

### Короткие — стандартные обозначения

Для общеизвестных инженерных величин — однобуквенные или короткие префиксы, совпадающие с символами IEC.

| Префикс | Величина | Типичные `units` |
| --- | --- | --- |
| `u_` | напряжение сети (AC mains, ~220В/380В, 50 Гц) | `V` |
| `v_` | низковольтное напряжение (DC, питание МК, плат, батарей) | `V`, `mV` |
| `i_` | ток | `A`, `mA` |
| `f_` | частота | `Hz` |
| `p_` | активная мощность | `W`, `kW` |
| `q_` | реактивная мощность | `var`, `kvar` |
| `s_` | полная мощность | `VA`, `kVA` |
| `e_` | энергия | `Wh`, `kWh`, `kvarh` |
| `t_` | температура | `deg C`, `deg F` |
| `pf_` | коэффициент мощности | — |
| `thd_` | THD | `%` |
| `co2_` | концентрация CO2 | `ppm` |
| `pm25_`, `pm10_`, `pm1_` | взвешенные частицы | `μg/m^3` |
| `voc_` | летучие органические | `ppm`, `ppb` |
| `cmd_` | команда (write-only) | — |

### Длинные — при отсутствии общепринятой короткой формы или при конфликте

| Префикс | Величина | Типичные `units` |
| --- | --- | --- |
| `time_` | длительность | `s`, `h`, `min` |
| `pressure_` | давление | `bar`, `Pa` |
| `volume_` | объём | `m^3`, `l` |
| `concentration_` | концентрация (общая) | `ppm`, `mg/m^3` |
| `count_` | счётчик событий | — |
| `state_` | дискретное состояние | — |
| `alarm_` | аварийный флаг | — |
| `level_` | уровень (заряд, шум, жидкость) | `%`, `m`, `dB` |
| `humidity_` | влажность | `%` |
| `illuminance_` | освещённость | `lx` |
| `sound_level_` | уровень звука | `dB` |
| `flow_` | расход | `m^3/h`, `l/min` |
| `speed_` | скорость | `m/s`, `rpm` |
| `angle_` | угол | `deg` |
| `resistance_` | сопротивление | `Ohm` |

**Разведение конфликтов однобуквенных префиксов:**

| Короткий | Длинный (используется длинный) |
| --- | --- |
| `u_` (сетевое напряжение AC) и `v_` (низковольтное DC) — оба используются раздельно | — |
| `t_` (температура) — используется | `time_` (время) — длинный, чтобы отличить от `t_` |
| `p_` (активная мощность) — используется | `pressure_` (давление) — длинный |
| `i_` (ток) — используется | `illuminance_` (освещённость) — длинный |
| `s_` (полная мощность) — используется | `speed_`, `sound_level_`, `state_` — длинные |
| `c_` — **не используется** (был бы конфликт count/co2/concentration/current) | `count_`, `co2_`, `concentration_` — каждое полное |
| `v_` (низковольтное DC) — используется | `volume_` (объём) — длинный |

**Префиксы парсятся как литеральные строки** до первого `_`: `t_mcu` → префикс `t`, `time_uptime` → префикс `time`. Они не путаются при программном разборе.

### Оговорка для нестандартных величин

Если измеряемая величина не входит в словарь — используется **понятное описательное имя** без специального префикса. Пример (WB-MSW):

| id | Что |
| --- | --- |
| `motion_current` | Текущее значение PIR-сенсора |
| `motion_max` | Максимум PIR с момента последнего сброса |

При появлении нового семейства устройств с нестандартными измерениями такие имена можно добавлять в словарь после согласования.

## Закрытый словарь модификаторов

| Модификатор | Где применяется |
| --- | --- |
| `min`, `max`, `avg` | `v_min_vin`, `t_max_mcu` |
| `peak`, `rms` | `u_rms_l1`, `u_peak_l1` (для сетевых `u_*` модификатор указывается явно) |
| `ap`, `an`, `rp`, `rn` | `e_ap_l1` (направление энергии: active +/−, reactive +/−) |
| `single`, `long`, `double`, `shortlong` | `count_single_in1` (виды нажатий) |

## Закрытый словарь сущностей

| Сущность | Значение |
| --- | --- |
| `vin` | клеммник входного питания (этикетка «Vin») |
| `input` | вход для UPS или преобразователя (если нет конкретного клеммника) |
| `mcu` | микроконтроллер |
| `board` | плата |
| `l1`, `l2`, `l3` | фаза по этикетке |
| `in<N>` | дискретный вход N |
| `k<N>` | релейный выход N |
| `ch<K>` | канал многоканального счётчика (`channel` → `ch`) |
| `total` | сумма по фазам / каналам |
| `ambient` | окружающая среда (для климатических датчиков) |
| `device` | устройство в целом |

**Оговорка:** если сущность не входит в словарь — используется понятное название (например, по этикетке устройства или по контексту).

## Что это даёт

1. **Короткие имена** для распространённых электрических величин: `v_mcu` вместо `mcu_voltage`, `i_rms_l1` вместо `irms_l1`.
2. **Сортировка по смыслу.** Низковольтные напряжения группируются как `v_*`, сетевые — как `u_*`, энергии — `e_*`, температуры — `t_*`. Полезно для UI, для grep, для AI-агента.
3. **Прохождение по §2.3 СТП 002 (≤ 4 единиц).** Все имена укладываются в лимит (раньше `min_mcu_voltage_since_startup` имел 5 единиц, теперь `v_min_mcu` — 3).
4. **Предсказуемость для AI-генератора.** Зная (величина, сущность), id выводится однозначно.
5. **Совместимость с инженерной нотацией IEC.** `V`, `I`, `P`, `Q`, `S`, `E` — стандартные символы.
6. **Существующие шаблоны не ломаются** — применяется только к новым.

## Что вызывает сомнение

1. **Явный модификатор `rms` в `u_rms_l1` / `i_rms_l1`.** RMS пишется явно — тогда читателю не нужно держать в голове, что для сетевых измерений по умолчанию подразумевается RMS. Для других режимов измерения тот же шаблон: `u_peak_l1`, `u_inst_l1`, `i_peak_l1`. Старая форма `urms_l1` (один токен `urms`) переходит в `u_rms_l1` (два токена через `_`), что соответствует общему шаблону `<величина>_<модификатор>_<сущность>`.
2. **`s_*` короткий, но `state_*` тоже на букву s.** При programmatic parsing различимы (s_l1 vs state_in1), для человека — небольшой когнитивный диссонанс.
3. **`p_*` (active power) vs `pressure_*`.** Используются на разных устройствах (счётчик мощности vs климат-датчик), но в смешанных шаблонах нужно внимательно. Для климат-датчика `p_ambient` мог бы означать «давление окружающее», но по словарю — «активная мощность окружающая», что бессмысленно. Правило: префикс читается строго по словарю, без выбора по контексту.

## Открытые вопросы для команды

1. **Релейные выходы:** `state_k1` (унификация) или оставить `k1` (устоявшаяся форма на этикетке)? Связанный вопрос для дискретных входов: `state_in1` или оставить `input_1`?

2. **`reset_all_counters` → `cmd_reset_counts`** — действительно унифицируем все команды через `cmd_`, или оставляем естественные имена?

3. **Канонический реестр в СТП 001:** заменяем существующие Приложения А, В, Г на новые имена (для новых шаблонов), или вводим отдельный раздел «Канонический реестр-2026»?

4. **Стандартизуем `motion_current`/`motion_max`** под общий шаблон, или принимаем их как ad-hoc по правилу «если нет в словаре — описательное имя»?

5. **Длина имён MAP-серии** (`ch<K>_u_rms_l<N>`, `ch<K>_e_ap_l<N>` и др.) — ровно 4 единицы, на пределе §2.3 СТП 002. Это нормально, или ослабить §2.3 (например, «технические индексы вроде `<K>` и `<N>` не учитываются при подсчёте»)?

## Применение

После согласования:

1. Обновить Приложения А, В, Г СТП 001-2025 изд.03 на новые имена.
2. Записать в `Открытые вопросы и упущения.md` решение по §2.3 СТП 002 (смягчение или нет).
3. Прописать в Jinja-макросах `templates/_macros/*.j2` новые имена (для новых шаблонов).
4. В существующие шаблоны изменения **не вносятся** — оставлены для будущей единичной миграции по решению команды.
