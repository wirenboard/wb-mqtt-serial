# Сравнение четырёх документов о шаблонах wb-mqtt-serial

## Назначение

Поставлена задача оценить, как соотносятся между собой нормативные и руководящие документы Wiren Board, касающиеся правил формирования идентификаторов, наименований и UI-текстов в шаблонах `wb-mqtt-serial`.

Документ — материал для синтеза итоговой редакции стандарта (изд.03).

## Главные противоречия между документами

1. **Сколько слов в id?**
   - «Текст в веб»: **≤ 4 слова**.
   - Wiki: **3-5 слов**.
   - СТП 02: ≤ 64 символа, без счётчика слов.
   - **Нужно согласовать единое число**.

2. **`CO2` / `°C`:**
   - «Текст в веб» однозначно: в UI пишем `CO2`/`°C`, в id — нет.
   - СТП 02 ничего об этом не говорит. Без правила в id может попасть `co_2` или `co2`.
   - **Нужно зафиксировать**: id — `co2` (без подчёркивания), UI — `CO2`.

3. **«е» vs «ё»:**
   - «Текст в веб»: всегда «е».
   - СТП ничего не говорит.
   - В переводах WB-шаблонов реально встречается смесь («счётчик» vs «счетчик»).

## Wiki vs СТП — расхождение по контракту

Wiki написана как **how-to** для **внешнего** автора шаблонов. СТП — **нормативка** для внутренней разработки. Это разные жанры, и их не нужно объединять. Правильная конструкция:

- СТП — нормативный источник, на который ссылается Wiki.
- Wiki — quick-start с примерами, не дублирующий нормы.

## Что добавить в СТП изд.03

### Из «Текст в веб-интерфейсе»

1. **Новый раздел «Стиль текстов»** (или ссылка на этот документ как нормативная):
   - «е» вместо «ё»
   - ASCII-кавычки `"`
   - Дефис вместо тире
   - Десятичный разделитель — точка

2. **Уточнить §7.4 «Регистр»:**
   - EN — Title Case (каждое слово с большой)
   - RU — Sentence case (только первое слово), исключение — аббревиатуры

3. **Уточнить §10.6 «Единицы»:**
   - В `title` — в круглых скобках после названия (`Debounce Time (ms)`)
   - В `name`/`id` — НИКОГДА; для этого поле `units`

4. **Новый раздел «Описания (`description`)»:**
   - 2-3 предложения максимум
   - Без точки в конце
   - Без CAPS LOCK (исключение — аббревиатуры)
   - Без скобок (раскрыть, если возможно)
   - Предпочтительно — название параметра самоочевидно, `description` не нужен

5. **Новое приложение «Особое написание в UI»:**
   - `CO2`, `°C`, `°F`, `m²`, `m³`, `Ω`, `μ`
   - Соответствующие id: `co2`, `degc`, без спецсимволов

### Из Wiki

6. **§6.2 «Длина идентификатора»** — добавить рекомендацию «3-5 слов» (Wiki) или явно «не более 4» («Текст в веб»). Согласовать единое число.

7. **Приложение З «Анти-примеры»** — расширить таблицей «Неправильно → Правильно» из Wiki:
   - `Status of Compressor 1` → `compressor_1_status`
   - `Ram Used` → `ram_used`
   - `VALVE_1_OUT` → `valve_1_out`
   - `Valve, Water` → `valve_water`

8. **§11.7 «Раздельные адреса»** — добавить второй случай: `on_value`/`off_value` для нестандартных бинарных представлений (`0x00a5` / `0x005a`).

## Резюме

- **«Текст в веб-интерфейсе»** — отличный стиль-гайд, **сильнее СТП в области UI-копирайта**. Должен либо стать нормативной ссылкой в СТП, либо быть инкорпорирован разделом.
- **Wiki** — полезная how-to, **частично перекрывает** СТП, должна остаться quick-start'ом со ссылкой на нормативный СТП.
- **СТП 01** — слабый и в стиле, и в нормативке.
- **СТП 02** — нормативно полнее всех, но **пробуксовывает в стилистике** UI-копирайта. Нужны дополнения из «Текст в веб» (правила «е/ё», регистр EN/RU, единицы, описания, `CO2`/`°C`, десятичный разделитель).
