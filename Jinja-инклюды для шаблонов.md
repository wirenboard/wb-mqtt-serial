# Применимость Jinja-инклюдов для централизации одинаковых сущностей в шаблонах wb-mqtt-serial

## Контекст

В шаблонах Wiren Board (112 файлов с `group: g-wb`) одни и те же сущности — `sn`, `fw`, `uptime`, `mcu_temperature`, `supply_voltage`, группы `g_hw_info`/`g_general`/`g_debug`, счётчики нажатий, входы/выходы — повторяются в десятках файлов с расхождениями в идентификаторах и переводах (см. отдельный документ «Стандартизация сущностей WB-шаблонов»). Поставлен вопрос: можно ли вынести их в одно место и подключать через Jinja-инклюды?

**Краткий ответ: да, можно, и это даст значимый выигрыш. Архитектурно — без изменений драйвера, только в сборке.**

## Как сейчас работают Jinja-шаблоны в проекте

Из `Makefile`:

```makefile
TEMPLATES_DIR = templates
JINJA_TEMPLATES = $(wildcard $(TEMPLATES_DIR)/*.json.jinja)

$(GENERATED_TEMPLATES_DIR)/%.json: $(TEMPLATES_DIR)/%.json.jinja
	mkdir -p $(GENERATED_TEMPLATES_DIR)
	(cd $(TEMPLATES_DIR); j2 -o ../$@ $(notdir $^))
```

**Ключевые факты:**

- Рендеринг — на **этапе сборки**, инструментом `j2` (jinja2-cli, Python). Не runtime.
- Драйвер `wb-mqtt-serial` видит уже готовый монолитный `.json` в `/usr/share/wb-mqtt-serial/templates/`.
- В runtime никакая инфраструктура для инклюдов не нужна.

Следствие: **любая Jinja-механика — инклюды, макросы, наследование, внешние данные — допустима без изменений драйвера**. Только сборка и пакетирование.

## Что предлагает Jinja2

Доступны четыре механизма для переиспользования (всё поддерживается в `j2` CLI):

1. `{% include "file.j2" %}` — вставка отрендеренного содержимого в точке вызова.
2. `{% from "file.j2" import macro %}` + вызов `{{ macro(args) }}` — макросы с параметрами.
3. `{% extends "base.j2" %}` + `{% block name %}` — наследование (классическая HTML-сборка).
4. **Внешние данные** — `j2 template.j2 data.yaml` подгружает структурированные данные, шаблон обращается к ним как к Jinja-переменным.

Применимо для нашего кейса: **2 и 4** — почти идеальная пара.

## Прототипы (проверены на этой машине)

### Подход А. Простые инклюды

```jinja
{# templates/_includes/_service_translations_en.j2 #}
"sn": "Serial Number",
"fw": "Firmware Version"
```

```jinja
{# templates/config-wb-foo.json.jinja #}
"translations": {
    "en": {
        "main_channel": "Main Channel",
        {% include "_includes/_service_translations_en.j2" %}
    }
}
```

**Минусы.**
- Запятые. Включаемый файл «не знает», в каком контексте он окажется. Хвостовая запятая в JSON — не valid. Нужно либо у самого включаемого файла дисциплина (последний элемент без запятой), либо у вызывающего шаблона.
- Разделение на `_en.j2` и `_ru.j2` — нужно держать строго в синхроне ключи.

### Подход Б. Макросы с параметрами (рекомендуемый)

```jinja
{# templates/_macros/wb_service.j2 #}
{%- macro channel_sn(address=270) -%}
{
    "name": "sn",
    "type": "text",
    "reg_type": "input",
    "address": {{ address }},
    "format": "u32",
    "group": "g_hw_info"
}
{%- endmacro %}

{%- set service_translations = {
    "sn":              {"en": "Serial Number",    "ru": "Серийный номер"},
    "fw":              {"en": "Firmware Version", "ru": "Версия прошивки"},
    "uptime":          {"en": "Uptime",           "ru": "Время работы с момента включения"},
    "mcu_temperature": {"en": "MCU Temperature",  "ru": "Температура МК"},
    "mcu_voltage":     {"en": "MCU Voltage",      "ru": "Напряжение МК"},
    "supply_voltage":  {"en": "Supply Voltage",   "ru": "Напряжение питания"},
    "g_hw_info":       {"en": "HW Info",          "ru": "Данные модуля"}
} %}

{%- macro translations_for(lang, keys) -%}
{%- for k in keys -%}
"{{ k }}": "{{ service_translations[k][lang] }}"
{%- if not loop.last %},
                {% endif -%}
{%- endfor -%}
{%- endmacro %}
```

```jinja
{# templates/config-wb-foo.json.jinja #}
{%- from "_macros/wb_service.j2" import channel_sn, channel_fw, translations_for -%}
{%- set service_keys = ["sn", "fw", "uptime", "g_hw_info", "g_general"] -%}
{
    "device": {
        "channels": [
            { "name": "main", "address": 0 },
            {{ channel_sn() }},
            {{ channel_fw(address=200) }}
        ],
        "translations": {
            "en": {
                "main": "Main",
                {{ translations_for("en", service_keys) }}
            },
            "ru": {
                "main": "Основной",
                {{ translations_for("ru", service_keys) }}
            }
        }
    }
}
```

**Прототип отработал** — на выходе валидный JSON с правильно расставленными запятыми, все 6 заявленных ключей переводов попали в обе языковые секции.

### Подход В. Внешние данные (YAML + Jinja-цикл)

```yaml
# templates/_data/canonical.yaml
service_translations:
  sn:              {en: "Serial Number",    ru: "Серийный номер"}
  fw:              {en: "Firmware Version", ru: "Версия прошивки"}
  uptime:          {en: "Uptime",           ru: "Время работы с момента включения"}
```

```jinja
{# templates/config-wb-foo.json.jinja #}
"translations": {
    {% for lang in ["en", "ru"] -%}
    "{{ lang }}": {
        {% for k, v in service_translations.items() -%}
        "{{ k }}": "{{ v[lang] }}"{% if not loop.last %},{% endif %}
        {% endfor %}
    }{% if not loop.last %},{% endif %}
    {% endfor %}
}
```

```makefile
j2 -f yaml template.json.jinja canonical.yaml > out.json
```

**Прототип отработал.** Все переводы из YAML корректно растеклись по обоим языкам.

## Сравнение подходов

| Критерий | А (include) | Б (macros) | В (внешний YAML) |
| --- | --- | --- | --- |
| Простота для автора шаблона | ✓ простая | ✗ нужны параметры | ✓ ничего, кроме переменных |
| Параметризация (адрес и т. п.) | ✗ нет | ✓ есть | ✓ через цикл |
| Запятые в JSON | ✗ ручные | ✓ внутри макроса | ✓ внутри цикла |
| Один источник переводов | ✓ можно | ✓ можно | ✓ строго |
| Возможность миксовать data + код | ✗ нет | ✓ есть (макрос + dict) | △ только данные |
| Внешняя зависимость в Makefile | нет | нет | надо передавать `-f yaml data.yaml` |
| Диагностика ошибок | средне | средне | средне |
| Удобство чтения PR | хорошо | хорошо | отлично (отдельный YAML) |

**Рекомендация:** **гибрид Б + В.**

- Канонический YAML/JSON-файл с переводами и метаданными (`address` по умолчанию, `units`, `format`) — единый источник.
- Набор макросов в `_macros/wb_service.j2`, которые читают эту структуру и эмитят правильно отформатированный JSON-фрагмент.
- Шаблоны устройств вызывают макросы.

## Что можно вынести в инклюды/макросы — оценка по типам сущностей

| Сущность | Применимость | Параметры макроса | Выигрыш |
| --- | --- | --- | --- |
| `sn` (Serial) | очень высокая | `address` (default 270) | Высокий — 79 шаблонов используют |
| `fw` (FW Version) | очень высокая | `address` (default 250) | Высокий — 60 шаблонов |
| `uptime` | очень высокая | `address` (default 104) | Высокий — 88 шаблонов |
| `mcu_temperature` | очень высокая | `address` (default 124) | Высокий — 51 шаблон |
| `mcu_voltage` | очень высокая | `address` (default 123) | Высокий — 51 шаблон |
| `supply_voltage` | очень высокая | `address` | Высокий — 58 шаблонов |
| `min_supply_voltage_since_startup` | высокая | `address` | Средний — 21 шаблон |
| `hw_batch_number` | высокая | `address` (default 304) | Средний — 30 шаблонов |
| Группы `g_hw_info`, `g_general`, `g_debug`, `g_safety` | очень высокая | — | Высокий — встречаются в 80%+ |
| Press counters (4 типа на вход) | очень высокая | `input_num`, `base_address` | Очень высокий — 4 канала × N входов |
| Каналы входов (`Input N`) | высокая | `input_num`, `address` | Высокий |
| Релейные выходы (`KN`) | высокая | `output_num`, `address` | Средний — кейсы разнообразнее |
| Safety mode параметры (несколько на выход) | средняя | вложенные параметры | Высокий, но макрос длинный |
| Translations.en/ru с одинаковым набором | очень высокая | список ключей | Очень высокий |
| Mapping matrix WB-MR (input × output × тип нажатия) | средняя | — | Средний, уже компактно через циклы |
| 3-фазные электрические каналы (`urms_lN`, `irms_lN`) | высокая | `phase`, базовый адрес | Высокий для счётчиков MAP |

**Оценка экономии строк:** для типичного WB-шаблона (`config-wb-mdm3.json.jinja`, ~600 строк) можно ожидать **−15…20 %** объёма за счёт служебных каналов и групп. Для WB-MR6Cv3 (1307 строк) выигрыш меньше — там уже массивно используются Jinja-циклы по входам и выходам, но вынос press counter в макрос всё равно убирает повторы.

## Архитектурное предложение

### Файловая структура

```
templates/
  _macros/                      ← НЕ устанавливается в /usr/share
    wb_service.j2               ← макросы для sn, fw, uptime и пр.
    wb_press_counters.j2        ← макросы для счётчиков нажатий
    wb_safety.j2                ← макросы для safety mode
  _data/                        ← НЕ устанавливается
    canonical.yaml              ← словарь переводов и метаданных
  config-wb-mr6cv3.json.jinja
  config-wb-mdm3.json.jinja
  ...
```

### Изменения в Makefile

```makefile
# 1. Wildcard *.json.jinja уже игнорирует _macros/*.j2 и _data/*.yaml — менять не нужно.

# 2. Добавить зависимость каждого .json.jinja от макросов и данных,
#    чтобы изменения в _macros/ или _data/ триггерили пересборку.
TEMPLATE_INCLUDES = $(wildcard $(TEMPLATES_DIR)/_macros/*.j2) \
                    $(wildcard $(TEMPLATES_DIR)/_data/*.yaml)

$(GENERATED_TEMPLATES_DIR)/%.json: $(TEMPLATES_DIR)/%.json.jinja $(TEMPLATE_INCLUDES)
	mkdir -p $(GENERATED_TEMPLATES_DIR)
	(cd $(TEMPLATES_DIR); j2 -f yaml -o ../$@ $(notdir $<) _data/canonical.yaml)
```

### Изменения в `debian/wb-mqtt-serial.install` (или `Makefile install`)

Убедиться, что в пакет попадают **только сгенерированные .json**, и НЕ попадают `_macros/`, `_data/`. В существующем `Makefile install` стоит обновить путь, если он сейчас копирует целиком `templates/`. Текущий `install` рекурсивно копирует только `$(GENERATED_TEMPLATES_DIR)/*.json` — поэтому скорее всего ничего не нужно.

## Риски и нюансы

### 1. Запятые в JSON

JSON не допускает trailing commas. При работе с массивами/объектами через `{% for %}` или макросы нужно проследить, чтобы:

- последний элемент массива не имел хвостовой запятой;
- последний ключ объекта переводов не имел хвостовой запятой.

В Jinja для этого стандартный трюк — `{% if not loop.last %},{% endif %}`. В прототипе он работает.

**Однако** в существующих шаблонах wb-mqtt-serial встречается **смягчённый JSON-парсер с поддержкой комментариев** (см. `dump_templates.py`, `strip_json_comments`). Поведение драйвера в C++ — стандартное (valijson), trailing commas не разрешает. Поэтому строгая дисциплина запятых обязательна.

### 2. Зависимости make

Make должен пересобирать шаблон при изменении макроса или данных. Без явных зависимостей мейк подумает, что `.json` свежий. Решено добавлением `$(TEMPLATE_INCLUDES)` в правило.

### 3. Диагностика ошибок Jinja

Ошибки в макросе показываются с указанием строки в макросе, но контекст вызова из шаблона часто нужно реконструировать вручную. Это терпимо при `j2 --no-undefined` и аккуратном именовании.

### 4. Сборка только jinja-шаблонов

Из 112 WB-шаблонов **только часть на Jinja**, остальные — `.json`. Чтобы получить выгоду от макросов на 100 % шаблонов, нужно сначала **сконвертировать оставшиеся `.json` в `.json.jinja`** (даже без активного использования Jinja-фич). Это рутинное переименование с символическими ссылками для совместимости — или одноразовая миграция.

Из `find templates -name "config-wb-*.json" | wc -l` (без `.jinja`) — это десятки файлов. См. ниже.

### 5. PR-ревью

Diff в PR показывает изменения **в исходных файлах** (jinja). Чтобы увидеть итоговый JSON, нужно либо локально собрать (`make templates`), либо смотреть артефакты CI. Это слегка увеличивает порог входа для ревьюера, но не блокер. Можно настроить CI на публикацию рендеренных артефактов.

### 6. Совместимость с внешними инструментами

- `dump_templates.py` (есть в репо) — читает JSON, не Jinja. Уже работает с готовыми сгенерированными `.json`.
- `TDeviceTemplatesTest.Validate` — тестирует уже сгенерированные шаблоны. Не затронут.
- Веб-конфигуратор — никогда не видит Jinja. Не затронут.

### 7. Конфликт с пользовательскими шаблонами

Пользовательские шаблоны лежат в `/etc/wb-mqtt-serial.conf.d/templates/`. Они **не проходят через сборку** — пользователь пишет .json вручную. Поэтому пользовательские шаблоны Jinja-инклюдов не имеют, и это для них норма.

Возможное расширение: если пользователю нужны те же макросы, мы могли бы документировать «как самому собрать шаблон с j2» — но это уже отдельная история.

## Сравнение объёма для конкретного шаблона

Для условного WB-устройства со «стандартным набором» (`sn`, `fw`, `uptime`, `mcu_temperature`, `mcu_voltage`, `supply_voltage`, `hw_batch_number`, плюс группа `g_hw_info`):

| Элемент | Объём «до» (строки) | Объём «после» (строки) |
| --- | --- | --- |
| Описание 7 каналов | ~70 | 7 (`{{ channel_sn() }}` и т. п.) |
| Группа `g_hw_info` | 4 | 1 (`{{ group_g_hw_info() }}`) |
| Переводы EN (7 ключей) | 7 строк | 1 строка (`{{ translations_for("en", keys) }}`) |
| Переводы RU (7 ключей) | 7 строк | 1 строка |
| Список ключей | — | 1 строка |
| **Итого** | **~88** | **~11** |

Это **−87 %** по строкам, без потери семантики. Распространим на 80 шаблонов — получаем существенное снижение технического долга.

## Дальнейшие шаги

Если решаем внедрять, разумная последовательность:

1. **Принять стандарт изд.02** (или его доработанную версию). Канонические идентификаторы должны быть зафиксированы перед макросами.
2. **Завести `templates/_macros/wb_service.j2` и `templates/_data/canonical.yaml`** с переводами для служебных каналов и общих групп (`sn`, `fw`, `uptime`, `mcu_*`, `supply_voltage`, `g_hw_info`, `g_general`, `g_debug`, `g_safety`).
3. **Обновить Makefile** — зависимость от макросов и YAML, передача YAML в j2.
4. **Мигрировать 1-2 эталонных шаблона** (например, `config-wb-ups-v3.json.jinja` и `config-wb-mr6cv3.json.jinja`) на макросы, чтобы обкатать механизм.
5. **Прогнать через `make test`** — `TDeviceTemplatesTest.Validate` проверит, что сгенерированный JSON совместим со схемой.
6. **Спланировать массовую миграцию** — отдельный проект, по группам устройств. Возможно — автоматизированный скрипт, делающий «глупую» замену по выявленным паттернам.

## Резюме

- **Технически осуществимо без изменения драйвера.** Все механизмы Jinja2 — макросы, инклюды, внешние данные — работают на этапе сборки через `j2` CLI, который уже используется в проекте.
- **Подход «макросы + внешний YAML»** даёт максимальный выигрыш: один источник для переводов и метаданных, параметризация под особенности устройства, корректное обращение со скобками и запятыми.
- **Экономия объёма** для типичного WB-шаблона со стандартным набором — **5-10× по строкам служебной части**. Точечно для шаблонов с массовым повтором (счётчики нажатий, входы/выходы) — ещё выше.
- **Главное достоинство — не объём, а единый источник истины.** Исправил `Серийный номер` → `S/N` в `canonical.yaml` — все шаблоны обновлены при следующей сборке. В нынешней практике это правится в 79 файлах вручную.
- **Главный риск — миграция.** Существующие 112 WB-шаблонов нужно переписать с использованием макросов, и часть из них сейчас вообще не `.jinja`. Стоит запланировать как отдельный проект, не привязанный к стандарту.
- **Стандарт изд.02 — предусловие** для адекватного `canonical.yaml`. Канонические имена должны быть согласованы до того, как они захардкожены в общий ресурс.

**Рекомендация:** включить в стандарт изд.03 раздел «Использование Jinja-макросов для общих сущностей» как **рекомендованную** (не обязательную) практику для новых шаблонов и для миграций. Создать `templates/_macros/` и `templates/_data/canonical.yaml` сразу при принятии стандарта.
