# User templates (метка «пользовательский» и RPC управления) — plan

## Scenarios

1. В выпадающем списке типов устройств веб-интерфейса пользовательские шаблоны (из `/etc/wb-mqtt-serial.conf.d/templates`) помечены бейджем «пользовательский» — и у пунктов раскрытого списка, и у выбранного значения. (Отрисовка — homeui, см. секцию «Сценарии и требования для homeui».)
2. wb-mqtt-serial в ответе `config/Load` сообщает для каждого типа устройства признак «пользовательский шаблон»: он установлен, когда действующий (предпочтительный) шаблон типа загружен из пользовательской директории — и для новых типов, и для перекрывающих штатные.
3. Через MQTT RPC можно загрузить файл шаблона на контроллер: содержимое валидируется и сохраняется в пользовательскую директорию; список типов и схемы редактора актуальны уже к моменту ответа. Невалидный шаблон отклоняется с внятной ошибкой, файловая система не меняется.
4. Если тип загружаемого шаблона используется устройствами текущей конфигурации, загрузка без явного подтверждения (`force`) отклоняется: их настройки построены по действующему шаблону, и перекрытие должно быть осознанным. С `force` шаблон записывается.
5. Загрузка/удаление шаблона не влияет на запущенный опрос: он продолжается по прежней паре «шаблон + конфигурация» (внутренне согласованной). Применение к опросу — штатным путём: сохранение конфигурации в редакторе, confed перезапускает сервис (как и сегодня при ручном изменении шаблонов по scp).
6. Через MQTT RPC можно удалить пользовательский шаблон по типу устройства: если он перекрывал штатный — тип возвращается к штатному шаблону; если был единственным — тип исчезает из списка. Штатные шаблоны через RPC удалить нельзя.
7. Группа типа в выпадающем списке определяется действующим шаблоном. Перекрывающий пользовательский шаблон может объявлять другую группу (или не объявлять её вовсе — тогда тип попадает в группу «Произвольные устройства» (группа `g-custom`), как и все шаблоны без группы) — при загрузке тип «переезжает» в неё из группы штатного шаблона. После удаления перекрытия тип возвращается в группу штатного шаблона.
8. Ручная правка/копирование файлов в `/etc/…/templates` (scp, редактор) подхватывается на лету, как и раньше; признак при этом остаётся корректным.
9. Перемещение файлов (`mv`) теперь тоже подхватывается на лету — в обеих директориях шаблонов: перемещение файла в директорию равносильно появлению/обновлению шаблона, перемещение из директории (и переименование в не-`.json`) — удалению. Сейчас вотчер такие события не видит — это дыра, в которую попадает типичный сценарий «скачал шаблон, сделал `mv` на контроллере».

## Contracts

Формат обмена — стандартный WB MQTT-RPC: запрос `{"id": N, "params": {…}}` в топик `/rpc/v1/wb-mqtt-serial/<сервис>/<метод>/<client_id>`, ответ в `…/reply` — `{"id": N, "result": {…}}` либо `{"id": N, "error": {"code": …, "message": …, "data": "<человекочитаемая причина>"}}`.

Ошибки — по [спецификации MQTT-RPC](https://github.com/wirenboard/mqtt-rpc): `{"id", "error": {"message", "code", "data"?}}`, где `data` — строковый тип ошибки для машинного различения (в примере спецификации — `"data": "ErrorType"`). Хендлер в wb-mqtt-serial управляет именно содержимым `data`; `message` wblib задаёт по коду («Server error»). Ошибка «тип используется устройствами текущей конфигурации, требуется `force`» (у `templates/Upload` и `templates/Delete`):

```json
{ "error": { "code": -32000, "message": "Server error", "data": "template-in-use" } }
```

Клиент распознаёт её по `error.data === "template-in-use"`; подробности (тип устройства, количество использующих устройств) сервис пишет в журнал. Прочие ошибки — как у существующих методов сервиса: человекочитаемая причина строкой в `data`.

- `config/Load` (существующий) — в элементах `types[].types[]` новое опциональное поле `"user-defined": true`: присутствует только у типов, чей действующий шаблон лежит в пользовательской директории. У штатных типов и псевдотипов протоколов (`"type": "protocol:*"`) поле не передаётся, отсутствие трактуется как `false` (экономия трафика: штатных типов сотни, аналогично опциональному `hw`). Остальные поля без изменений.
- `templates/Upload` (новый)
  - Запрос: `{ "content": string, "filename": string, "lang": string?, "force": boolean? }` — `content`: полный текст файла шаблона (JSON), непустой; `filename` (обязательный): имя файла, непустое — сервер берёт basename, заменяет символы вне `[A-Za-z0-9._-]` на `_` и гарантирует расширение `.json`, выход за пределы пользовательской директории невозможен; `lang`: язык локализованных полей ответа (как у `Load`, по умолчанию `en`); `force`: подтверждение перекрытия используемого типа (по умолчанию `false`).
  - Ответ: `{ "types": [ … ] }` — тот же формат, что поле `types` в ответе `Load`: массив групп, здесь всегда одна группа с одним типом. Пример:

    ```json
    { "types": [ { "name": "Произвольные устройства", "types": [ { "name": "Мой контроллер", "type": "my_device", "deprecated": false, "protocol": "modbus", "mqtt-id": "my-device", "with-subdevices": false, "user-defined": true } ] } ] }
    ```

    `name` группы — переведённое имя (по `lang`), как в `Load`; в описании загруженного типа `"user-defined": true` присутствует всегда. Группа в ответе — группа **действующего** (загруженного) шаблона: при перекрытии штатного типа она может отличаться от прежней (тип переезжает между группами списка; шаблон без группы попадает в «Произвольные устройства» (группа `g-custom`)).
  - Поведение: валидация (синтаксис JSON → наличие непустого `device_type` → полная проверка шаблона, та же, что выполняется при его первом использовании: JSON-схема, разбор `condition`-выражений, согласованность адресов параметров, вложенность subdevices; шаблоны с `"deprecated": true` отклоняются — при загрузке с диска они освобождены от этих проверок, поэтому их приём через RPC не поддерживается; любой отклонённый шаблон гарантированно не меняет файловую систему) → если `device_type` используется устройствами текущей конфигурации (`/etc/wb-mqtt-serial.conf`) и `force` не задан — ошибка, файл не записывается → запись в пользовательскую директорию под санитизированным `filename`. Инвариант «один файл на тип»: если у типа уже был пользовательский шаблон в файле с другим именем — старый файл удаляется, дубликата не остаётся. Если имя занято файлом другого типа — добавляется числовой суффикс (`<имя>-1.json`, `-2`, …), чужие файлы не затираются → список типов и схемы обновлены уже к моменту ответа.
  - Ошибки: тип используется устройствами текущей конфигурации, а `force` не задан — `data: "template-in-use"` (подробности в журнале сервиса); остальные — текстовая причина в `data`: запрос не по схеме; некорректный JSON; отсутствует/пуст `device_type`; шаблон помечен `"deprecated"`; шаблон не проходит схему; ошибка записи.
- `templates/Delete` (новый)
  - Запрос: `{ "type": string, "lang": string?, "force": boolean? }` — `force`: подтверждение удаления шаблона используемого типа (по умолчанию `false`), симметрично `Upload`; `lang` — как у `Upload`.
  - Ответ: `{ "types": [ … ] }` — как у `Upload`: если тип остался (удалено перекрытие — действующим снова стал штатный шаблон) — одна группа с описанием нового действующего типа (уже без `"user-defined"`; группа — штатного шаблона, может отличаться от группы удалённого пользовательского); если тип исчез совсем (пользовательский шаблон был единственным) — пустой массив `"types": []`.
  - Поведение: если тип используется устройствами текущей конфигурации и `force` не задан — ошибка, файл не удаляется; иначе удаляет файл действующего пользовательского шаблона данного типа; список типов и схемы обновлены к моменту ответа.
  - Ошибки: у типа нет пользовательского шаблона (неизвестный тип, только штатный, повторное удаление) — текстовая причина; тип используется устройствами текущей конфигурации, а `force` не задан — `data: "template-in-use"` (как у `Upload`); ошибка удаления файла — текстовая причина.
- Формат файлов шаблонов не меняется: признак выводится из расположения файла, нового ключа в шаблонах нет.

## Сценарии и требования для homeui (отдельная задача)

Реализуется отдельной задачей в репозитории homeui; требования фиксируются здесь, бэкенд-контракт выше.

1. **Бейдж**: «пользовательский» (en: user-defined) у типа с `"user-defined": true` (отсутствие поля = штатный) — в пункте раскрытого списка и у выбранного значения селектора типа устройства.
2. **Загрузка шаблона**: кнопка в редакторе конфигурации serial-устройств, выбор локального `.json`-файла → `templates/Upload {content, filename: <имя выбранного файла>, lang}`.
3. **Обработка ответа Upload/Delete** (формат `types` из `Load`), слияние в стор типов с учётом смены группы:
   - найти тип по `type` во **всех** группах стора и удалить прежнюю запись (группа могла измениться);
   - вставить описание из ответа в группу с именем из ответа: группы нет — создать (алфавитный порядок групп после блока Wiren Board, как строит `Load`); внутри группы — по алфавиту имён типов;
   - если прежняя группа опустела — убрать её из списка;
   - повторный `config/Load` не нужен.
4. **Диалог перекрытия**: если `error.data === "template-in-use"` — показать подтверждение (количество затронутых устройств homeui считает по своему состоянию конфига; настройки будут отображаться по новому шаблону; к опросу применится после сохранения конфигурации) → при согласии повторить запрос с `"force": true`. Прочие ошибки — обычное сообщение об ошибке.
5. **Применение шаблона для используемого типа**: после успешной force-загрузки пометить конфигурацию изменённой — кнопка «Сохранить» активна, даже если настройки не правились, — и показать уведомление «шаблон применится к опросу после сохранения конфигурации». Сохранение штатно перезапускает сервис (в `wb-mqtt-serial-dummy.schema.json` объявлен `"service": "wb-mqtt-serial"`), опрос переходит на новый шаблон. Несовместимости настроек с новым шаблоном подсвечивает существующая валидация редактора.
6. **Удаление шаблона**: действие для типов с бейджем → `templates/Delete`; на ошибку с `error.data === "template-in-use"` — диалог подтверждения (как в п. 4, но про возврат на штатный шаблон) → повтор с `"force": true`. По ответу: непустой `types` → слить в стор, как в п. 3 (бейдж исчезает, `name`/`deprecated`/группа могли смениться); `"types": []` → удалить тип из стора. Для используемых типов — то же поведение применения, что в п. 5 (пометить изменённой, уведомление).
7. **Переводы** ru/en: бейдж, кнопка, диалоги, уведомления.
8. Техническое: текст шаблона в MQTT-payload передаётся JSON-экранированным (размер до ~2×); проверить ограничения MQTT-over-WebSocket на стороне UI.

## Test plan

- Существующие тесты должны проходить целиком (`make test`, gtest под valgrind), особенно: `TDeviceTemplatesTest.*`, `TConfigParserTest.*`, `TDeviceLoadConfigTest.*`.
- Новые (предлагаемые сьюты; фикстуры — «пользовательская» директория шаблонов рядом с существующей `test/device-templates/` и конфиг с устройством известного типа):
  - Признак (`TUserTemplatesTest`): перекрытие штатного типа пользовательским → признак установлен и действует пользовательский шаблон; новый пользовательский тип → установлен; штатный без перекрытия → снят; без заданной пользовательской директории → всегда снят; после живой перезагрузки файла признак корректен (для файла из пользовательской директории — установлен, из штатной — нет).
  - Загрузка (`TRPCTemplatesHandlerTest`): новый тип — файл создан, тип появился, ответ — группа с описанием типа в формате `types` из `Load`, с переведённым именем группы и `"user-defined": true`; повторная загрузка того же типа — файл перезаписан без дубликата; загрузка с `filename` → файл под этим именем, повторная загрузка типа с другим `filename` → старый файл удалён, тип один; `filename` с путём (`../x.json`, `/etc/passwd`) или спецсимволами → записан только санитизированный basename в пользовательской директории; тип используется в конфиге и `force` не задан → ошибка с `data: "template-in-use"`, файл не записан; то же с `"force": true` → записан; перекрытие неиспользуемого штатного типа — без force, штатный файл не тронут, действующим стал пользовательский; перекрытие шаблоном с другой/не указанной группой → в ответе группа действующего шаблона (для не указанной — «Произвольные устройства» (группа `g-custom`)); коллизия имени файла с чужим типом → создан файл с числовым суффиксом, оба типа живы; невалидный JSON / отсутствие `device_type` / нарушение схемы / битое `condition`-выражение → ошибка, в директории не осталось ни файла, ни временных остатков; шаблон с `"deprecated": true` отклоняется, файловая система не меняется.
  - Вотчер (`TFilesWatcherTest`, интеграционно на временной директории): `mv` файла в наблюдаемую директорию → колбэк «изменение» и шаблон появляется в карте; `mv` из директории → колбэк «удаление» и шаблон уходит из карты; переименование внутри директории → старое имя удалено, новое загружено; признак «пользовательский» после `mv` корректен.
  - Удаление: удаление перекрытия → возврат к штатному (включая инвалидацию кэша схем), в ответе группа с описанием штатного типа без `"user-defined"` (группа штатного шаблона, в т.ч. когда у пользовательского была другая); удаление единственного пользовательского типа → тип исчез, ответ `"types": []`; тип используется в конфиге и `force` не задан → ошибка с `data: "template-in-use"`, файл на месте; то же с `"force": true` → удалён; тип без пользовательского шаблона / повторное удаление → ошибка без падения.
- Ручная проверка на контроллере — минимальный интеграционный смоук поверх реального MQTT и запущенного сервиса (вариации логики покрыты авто-тестами, здесь проверяется обвязка: регистрация методов, установленные схемы, вотчер): через `mosquitto_rr` загрузить шаблон нового типа → в `result` группа с типом, файл появился в `/etc/wb-mqtt-serial.conf.d/templates`, `config/Load` показывает `"user-defined": true`; загрузить шаблон типа, используемого в конфиге, без `force` → `error.data === "template-in-use"`; удалить шаблон → тип вернулся к штатному.

## Out of scope

- homeui: реализация бейджа, кнопок, диалогов и уведомлений — отдельная задача по секции «Сценарии и требования для homeui».
- Формат файлов шаблонов и их JSON-схема.
- Автоперечитывание действующей конфигурации/перезапуск сервиса при изменении шаблона: применение к опросу — только штатным сохранением конфигурации.
- Управление штатными шаблонами (`/usr/share/wb-mqtt-serial/templates`) через RPC.
- Аутентификация/авторизация RPC — как у существующих методов сервиса.
