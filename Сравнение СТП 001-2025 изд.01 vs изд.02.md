# Сравнение СТП 001-2025 — изд.01 vs изд.02 (независимый черновик)

## Назначение документа

Документ построчно сравнивает два рабочих черновика стандарта на правила формирования идентификаторов и наименований в шаблонах драйвера `wb-mqtt-serial`:

- **Изд.01** — `СТП 001-2025 изд.01.md` (исходная редакция автора).
- **Изд.02** — `СТП 001-2025 изд.02 (независимый черновик).md` (редакция, написанная с чистого листа на основе анализа существующих шаблонов, JSON-схемы и отраслевых конвенций).

Цель сравнения — увидеть, где редакции совпадают, где расходятся и где одна редакция добавляет к другой. Документ нужен как материал для синтеза итоговой версии (изд.03).

## TL;DR — десять главных расхождений

| # | Тема | Изд.01 | Изд.02 |
| --- | --- | --- | --- |
| 1 | **Жанр** | смесь нормы и авторских заметок («Сейчас:», «Предлагаю отказаться») | строго нормативный текст + отдельные «Обоснования» |
| 2 | **Поле `device.id`** | сведено к идентификатору устройства, путаница с `device_type` | разведено явно: `device_type` (опаковый ключ конфига), `device.id` (MQTT-топик), `device.name` (отображение) |
| 3 | **Алфавит и дефис** | «только латиница строчная + `_`», но во всех примерах есть `-` | разрешено только `[a-z][a-z0-9_]*`; `-` исключение только для имён файлов; противоречие Изд.01 разрешено |
| 4 | **`enum_titles`** | идентификатор привязан к **индексу** элемента массива | привязка к **значению регистра** (устойчивость к перестановке `enum`) |
| 5 | **Битовые поля** | не упомянуты | отдельный раздел с уже существующим синтаксисом `address:shift:width`; каждый значимый бит — отдельный контрол |
| 6 | **Многорегистровые параметры, `word_order`** | не упомянуты | отдельный раздел; для 32/64-бит обязательно явное указание `word_order` |
| 7 | **Status/Control Word, VFD panel-коды, OBIS-счётчики** | не упомянуты | отдельные разделы 11.1, 11.3, 11.4 + Приложение Д с каноническими именами трёхфазных счётчиков |
| 8 | **Версионирование шаблона, `tpl<N>_` префикс, `-deprecated`** | упомянут только релиз `r01` | полная процедура миграции из README + критерии breaking/non-breaking |
| 9 | **Конформность, линтер, миграционная политика** | не упомянуты | отдельные разделы 14 и 15 |
| 10 | **Приложения** | мини-таблица из двух строк (`sn`, `fw`) | 9 приложений: реестр групп, служебные каналы, канонические группы параметров, бренды, OBIS-имена, префиксы аварий, HA `device_class`, эталоны/анти-примеры, маппинг полей на разделы |

## Сводная оценка

| Критерий | Изд.01 | Изд.02 |
| --- | --- | --- |
| Объём | 401 строка (~50 КБ текста + ~120 КБ base64-картинки) | 1385 строк (~100 КБ текста, без иллюстраций) |
| Покрытие элементов JSON-схемы | ~50% | ~95% |
| Покрытие отраслевых сценариев (VFD, OBIS, Status Word, многоканальные) | 0% | ~90% |
| Готовность к публикации как нормативный документ СТП | низкая (требует чистки от заметок и достройки) | средняя (нужна верификация внешних ссылок, доработка приложений) |
| Готовность к программной проверке (линтер) | низкая (правила декларативны, без регекспов) | высокая (нормы формализованы через регекспы и закрытые словари) |
| Сохранение визуального контекста (UI) | высокая (скриншоты) | низкая (нет иллюстраций) |
| Учёт миграции существующих шаблонов | отсутствует | присутствует (раздел 15) |
| Учёт смешанной аудитории (WB + OEM + конечные пользователи) | низкий | средний |

## Рекомендация для итоговой редакции (изд.03)

Синтез не предмет данного документа, но направление очевидно:

1. **Структура** — взять из Изд.02 (полная структура СТП).
2. **Наглядность** — вернуть скриншоты из Изд.01 в соответствующие разделы Изд.02.
3. **Курсивные «Сейчас:»** — перенести в Приложение «Изменения относительно текущей практики» либо в Приложение З (анти-примеры).
4. **Решения по политически чувствительным вопросам** (группы устройств, deprecated, бренды) — взять из Изд.02, аргументацию автора Изд.01 — в обоснования.
5. **Все добавления Изд.02** (разделы 10, 11, 14, 15, приложения) — без изменений.
6. **Глоссарий** — взять из Изд.02 (расширенный).
7. **Перед публикацией** — обязательная верификация нормативных ссылок и URL спецификаций.
